# 文档管理模块说明文档

> 最后更新：2026-06-01

---

## 一、模块概述

文档管理模块是 AI 知识库管理平台的核心入口，负责将外部文档（文件上传）转化为结构化知识。完整流水线为：

```text
文件上传 → 解析文档 → 文本分段 → AI 提炼 → 候选审核 → 确认入库
```

### 核心能力a

- 支持 **11 种文件格式**上传解析
- 自动编码检测（GBK / GB2312 / BIG5 / SHIFT_JIS / UTF-8）
- **语义向量切分** + 机械切分回退
- AI 自动提炼 **5 种知识类型**（FAQ / 概念 / 流程 / 笔记 / 摘要）
- **候选审核工作台**，支持编辑、确认、拒绝和重试

## 三、功能说明

### 3.1 文件上传

- **入口**：`/documents` 页面顶部上传区域
- **支持格式**：`.txt` `.md` `.csv` `.xlsx` `.docx` `.pdf` `.png` `.jpg` `.jpeg` `.webp` `.bmp`
- **大小限制**：单文件最大 100MB
- **上传方式**：点击选择文件 或 拖拽到上传区域
- **多文件支持**：可同时上传多个文件，每个文件显示独立进度

上传后文档状态为 `uploaded`（待解析）。

### 3.2 文档解析

- **入口**：文档列表每行的「解析」按钮，或勾选多个后「批量解析」
- **解析流程**：
  1. 读取已保存的文件
  2. 自动检测文本编码（`jschardet` + `iconv-lite`）
  3. 调用对应解析器提取文本内容
  4. 语义向量切分（DashScope text-embedding-v4）
  5. 语义切分失败则回退到机械段落切分
  6. 保存切分结果到 `DocumentChunk` 表
- **状态变化**：`uploaded` → `parsing` → `parsed` / `failed`

解析成功后文档状态变为 `parsed`（已解析），可进行 AI 提炼。

### 3.3 文档预览

- **入口**：已解析文档的「预览」按钮
- **三个 Tab**：
  - **原始文本** — 显示完整解析文本，支持编辑
  - **分段视图** — 逐段展示切分结果
  - **提炼知识** — 展示已提取的候选知识，可编辑修改

### 3.4 AI 知识提炼

- **入口**：已解析文档的「提炼」按钮
- **提炼流程**：
  1. 逐段调用 LLM（使用 `OPENAI_API_KEY` / `OPENAI_BASE_URL` / `OPENAI_MODEL` 配置）
  2. 每段附加上下文信息（文档名、分段序号）
  3. LLM 返回 JSON 数组 → 5 层 JSON 解析容错
  4. 跨段去重（基于标题相似度）
  5. 写入 `CandidateKnowledge` 表
- **前置条件**：文档状态必须为 `parsed`

提炼完成后的知识进入候选审核状态。

### 3.5 候选审核

- **入口**：`/candidates` 审核工作台
- **操作**：
  - **确认**（批量）— 标记为 `confirmed`，后续可入库
  - **拒绝**（单个删除）— 从候选列表移除
  - **编辑** — 修改标题、内容、分类、标签、类型
  - **重试** — 删除该文档所有候选知识并重新提取

## 四、数据模型

### 4.1 文档上传流水线

```
DocumentSource (上传工作区)
  ├── id, originalName, fileType, fileSize
  ├── status: uploading → uploaded → parsing → parsed / failed
  ├── content (解析后的完整文本)
  ├── chunkCount, errorMessage
  └── chunks → DocumentChunk[]

DocumentChunk (文本切块)
  ├── documentSourceId → DocumentSource
  ├── chunkIndex, content
  ├── charStart, charEnd
  └── createdAt

CandidateKnowledge (AI 提取候选)
  ├── documentSourceId → DocumentSource (可选)
  ├── title, content
  ├── suggestedCategory, suggestedTags (JSON string)
  ├── type: faq | concept | procedure | note | summary
  ├── status: pending | confirmed | rejected
  └── createdAt, updatedAt
```

### 4.2 正式知识库（独立路径）

```
KnowledgeBase ──< KnowledgeBaseDocument >── KnowledgeDocument
                      (M:N 关联表)
KnowledgeDocument ──< KnowledgeChunk
```

两套模型当前**独立运行**：候选知识确认后，需手动或通过后续开发将确认的知识写入正式知识库。

## 八、已知问题

### 8.1 AI 提炼 422 错误

**现象**：点击「提炼」按钮后返回 422，提示提炼失败。

**可能原因**：

| 错误码 | 原因 | 解决方式 |
|--------|------|---------|
| `INVALID_STATUS` | 文档尚未解析（状态不是 `parsed`） | 先点击「解析」按钮，待解析完成后再提炼 |
| `EMPTY_DOCUMENT` | 解析后分段为空 | 检查文档是否有可读文本内容，或重新解析 |
| `EXTRACTION_FAILED` | LLM 调用全部失败（API Key 错误、网络不通、模型不可用） | 检查 `.env` 中 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL` 配置 |

**排查步骤**：
1. 确认文档状态为「已解析」（绿色标签）
2. 检查浏览器 Network 面板中 `/api/documents/[id]` 的返回，确认 `status: "parsed"`
3. 检查 `.env` 文件中 LLM 相关环境变量是否正确
4. 查看终端中服务端错误日志

### 8.2 解析图片文件需要 Vision API

图片文件（.png / .jpg / .jpeg / .webp / .bmp）的解析需要 Vision API 支持。如果 `VISION_API_KEY` 未配置或 Vision API 不可用，会回退到 Tesseract.js 本地 OCR（中文 + 英文），OCR 准确度有限。

### 8.3 语义分段依赖 DashScope

语义向量切分需要 `DASHSCOPE_API_KEY` 配置正确。如果 DashScope API 调用失败，会自动回退到机械段落切分，不影响基本功能，但分段质量可能下降。

### 8.4 候选知识确认后未自动入库

当前 `confirmCandidates` 仅将候选知识状态标记为 `confirmed`，**未自动写入正式知识库（KnowledgeDocument）**。确认后的知识需通过后续功能手动或自动关联到知识库。

### 8.5 大文件提炼耗时较长

AI 提炼逐段串行调用 LLM，如果一个文档被切分为 N 段，需要 N 次 LLM 调用，大文档可能耗时较长（每段约 2-10 秒）。前端暂无实时进度反馈。

### 8.6 编码检测可能不准确

短文本或混合编码文件的编码检测可能出错，导致中文乱码。可在预览 Tab 中手动编辑修正文本内容。

## 十一、使用流程示例

### 完整操作步骤

1. 访问 `http://localhost:3000/documents`
2. 上传一个 `.txt` 或 `.md` 文件
3. 文件出现在列表中，状态为「待解析」
4. 点击「解析」按钮，状态变为「解析中」→「已解析」
5. 点击「预览」查看分段结果
6. 点击「提炼」触发 AI 知识提取
7. 提炼完成后，点击「审核工作台」或文档行的「审核」按钮
8. 在 `/candidates` 页面审核候选知识：编辑、确认或拒绝
9. 确认的知识可用于后续入库操作
