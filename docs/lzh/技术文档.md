# AI 知识库管理平台技术文档

> 面向项目提交与面试答辩使用。本文档以 RAG 知识库管理、知识源归属、知识条目定位、召回 Debug 和知识库标签管理为主线，说明前端页面、接口服务、数据库结构和状态管理设计。

## 1. 模块定位

本项目是一个 AI 知识库管理平台，核心流程是将文档、笔记等知识源加工为可检索的知识片段，再交给 RAG 检索和 Agent 对话消费。本文档覆盖的重点不是单个页面的 CRUD，而是管理端如何把“知识库配置、文档归属、chunk 查看、召回调试、标签识别”串成一个可维护的闭环。

本部分主要交付能力包括：

- 后台统一布局：统一 Header、Sidebar、Main Content，承载知识库、笔记、文档、Agent 和对话页面。
- 知识笔记管理：支持 Markdown 笔记目录、编辑、保存、切换前自动保存、删除和知识源状态控制。
- RAG 知识库列表管理：创建、编辑、删除、搜索、排序、状态展示、标签展示。
- 知识库标签管理：知识库级标签绑定、新建、解绑、候选标签复用和无人使用标签自动清理。
- 知识库详情页：在同一路由内切换“知识条目”“文档管理”“Debug”三个视图。
- 文档归属管理：从可用知识源中选择文档绑定到当前知识库，并支持未保存变更提示。
- Chunk 查看与定位：查看文档切分后的知识片段，支持详情页搜索、筛选、跳转和高亮。
- RAG Debug：复用真实检索接口，展示召回 chunk、相关度得分、来源和 Prompt 上下文。

该模块位于知识生产和知识消费之间。上游负责文档上传、解析、切分和向量化；下游 Agent 对话和问答接口消费 RAG 检索结果。管理端的价值在于让知识库配置、文档归属和召回效果都可见、可查、可调。

## 2. 技术栈与分层

前端使用 Next.js App Router、React、TypeScript、Tailwind CSS、shadcn/ui、lucide-react 和 Zustand。页面组件保持轻量，复杂逻辑拆到 `src/features/knowledge-bases/`、`src/features/knowledge/`、`src/server/services/` 等模块。

后端使用 Next.js API Route、Zod、Prisma 7、SQLite。数据库访问统一通过 Prisma，服务层负责业务规则，Route Handler 只做请求解析、校验、调用服务和返回响应。

整体分层如下：

```text
Page Route
-> Feature Component
-> Feature API Wrapper
-> API Route
-> Zod Schema
-> Service
-> Prisma
-> SQLite
```

## 3. 数据模型设计

实际数据库结构以 `prisma/schema.prisma` 为准。RAG 管理相关核心模型如下。

### 3.1 KnowledgeBase

`KnowledgeBase` 表示一个 RAG 知识库，保存名称、描述、图标、颜色、状态和检索配置。

关键字段：

- `name`：知识库名称，唯一。
- `description`：知识库描述。
- `icon` / `color`：卡片展示使用。
- `topK`：当前可配置的召回结果数量。
- `similarityThreshold`：历史兼容字段，当前管理卡片、编辑弹窗和 Debug 左侧面板的展示重点是 TopK、状态和标签。
- `status`：`active` / `disabled`，控制知识库启用状态。
- `documents`：通过 `KnowledgeBaseDocument` 关联文档。
- `tags`：通过 `KnowledgeBaseTag` 关联知识库标签。

设计说明：`similarityThreshold` 作为历史兼容字段保留在数据结构中；前端管理视图聚焦当前真实生效的 TopK 和状态配置，避免管理参数和底层检索行为产生误读。

### 3.2 DocumentSource

`DocumentSource` 是统一知识源模型，承载上传文档、Markdown 笔记和其他可作为知识来源的内容。

关键字段：

- `title` / `originalName`：展示标题和原始名称。
- `fileType`：区分文件、笔记等类型。
- `sourceType`：记录来源，如 `manual`、`file`、`markdown`。
- `rawContent`：解析后的正文或 Markdown 正文。
- `status`：处理状态，如 `uploading`、`parsed`、`failed`。
- `activeStatus`：业务启用状态。
- `chunkCount`：冗余保存 chunk 数量，方便列表展示。
- `chunks`：关联 `DocumentChunk`。

设计说明：用一个统一的 `DocumentSource` 表承接不同来源，避免上传文档、笔记、知识源各自一套模型，后续只要满足 `status=parsed` 且 `activeStatus=active`，就可以被知识库引用。

### 3.3 KnowledgeBaseDocument

`KnowledgeBaseDocument` 是知识库和文档之间的多对多关系表。

关键规则：

- `@@unique([knowledgeBaseId, documentId])`：同一文档不能重复绑定到同一知识库。
- `onDelete: Cascade`：删除知识库或文档时自动删除关系。

设计说明：文档归属是“引用关系”，解绑知识库文档时只调整 `KnowledgeBaseDocument` 关系。这样同一份知识源可以被多个 RAG 知识库复用，文档本体和 chunk 仍作为可复用知识源保留。

### 3.4 DocumentChunk 与 ChunkEmbedding

`DocumentChunk` 保存文档切分后的知识片段，`ChunkEmbedding` 保存对应向量索引信息。

`DocumentChunk` 关键字段：

- `content`：chunk 正文。
- `chunkIndex`：文档内顺序。
- `chunkType`：`text` 或 `knowledge`。
- `title` / `suggestedCategory` / `suggestedTags`：AI 提炼字段。
- `reviewStatus`：知识型 chunk 的审核状态。
- `chunkStatus`：chunk 是否可用。

`ChunkEmbedding` 关键字段：

- `chunkId`：对应 chunk。
- `embedding`：向量内容。
- `embeddingModel`：生成向量的模型。
- `contentHash`：用于判断向量是否与当前 chunk 内容一致。

设计说明：`suggestedTags` 是 chunk 级 AI 建议标签，和本次新增的知识库级标签不是同一语义，不能混用。

### 3.5 KnowledgeTag 与 KnowledgeBaseTag

`KnowledgeTag` 是全局标签表，`KnowledgeBaseTag` 是知识库和标签之间的绑定表。

```text
KnowledgeTag
- id
- name
- color
- sortOrder
- createdAt
- updatedAt

KnowledgeBaseTag
- id
- knowledgeBaseId
- tagId
- createdAt
```

关键规则：

- 同一知识库不能重复绑定同一个标签。
- 删除知识库时级联删除标签绑定关系。
- 标签候选栏只展示至少被一个 RAG 使用的标签。
- 无人使用的标签由后端自动清理，候选栏保持精简。

### 4.1 后台统一布局

布局相关文件：

- `src/components/layout/admin-sidebar.tsx`
- `src/components/layout/admin-header.tsx`

后台采用固定的管理端骨架：

```text
AdminShell
-> AdminSidebar
-> Main Area
   -> AdminHeader

```

核心能力：

- `/` 默认重定向到 `/knowledge-bases`。
- Sidebar 统一承载知识库、知识笔记、知识文档、专家 Agent、专家对话等入口。
- Header 根据当前 pathname 派生页面标题。

- `/agents/chat` 独立作为专家对话入口，避免被 `/agents` 的前缀匹配误选中。

布局状态说明：

- `AdminShell` 当前实际使用本地 `sidebarOpen` state 控制侧边栏展开/收起。
- `src/store/slices/note.ts` 中预留了 `sidebarOpen`、`setSidebarOpen`、`toggleSidebar` 这组 AppSlice 能力，可用于后续布局级状态统一管理。
- 从状态边界看，Sidebar 展开状态属于布局级 UI 状态；当前本地 state 已能支撑 Header 与 Sidebar 联动，后续如需跨布局组件同步，可以统一接入 AppSlice。

### 4.2 知识笔记管理

入口文件：

- `src/app/note/page.tsx`
- `src/features/note/index.tsx`

组件拆分：

- `note-topbar.tsx`：标题、保存、新建、删除、知识源开关。
- `note-editor.tsx`：Markdown 编辑与预览。
- `note-directory.tsx`：右侧笔记目录。
- `note-empty-state.tsx`：空状态。
- `delete-note-dialog.tsx`：删除确认。

核心功能：

- 进入 `/note` 后加载笔记目录。
- 默认选中更新时间最新的笔记。
- 目录接口只返回摘要，不返回 `rawContent`。
- 选中笔记后按 id 加载详情正文。
- 支持 Markdown 正文编辑与实时预览。
- 支持标题编辑、保存、新建和删除。
- 切换笔记前自动保存当前草稿，保存失败时不切换。
- 右侧目录可收缩和展开。
- 空状态下提供创建入口。
- 知识源开关用于控制该笔记是否可作为 RAG 待选知识源。

相关接口：

| 功能         | 接口                     |
| ------------ | ------------------------ |
| 获取笔记目录 | `GET /api/notes`         |
| 创建笔记     | `POST /api/notes`        |
| 获取笔记详情 | `GET /api/notes/[id]`    |
| 更新笔记     | `PATCH /api/notes/[id]`  |
| 删除笔记     | `DELETE /api/notes/[id]` |

落库方式：

- 笔记复用 `DocumentSource`。
- `fileType = "note"`。
- `sourceType = "markdown"`。
- Markdown 正文保存到 `rawContent`。
- `fileSize` 根据正文 UTF-8 字节数计算。
- 新建笔记默认 `status = "pending"`。
- 当前知识源开关打开时提交 `status = "uploaded"`、`activeStatus = "active"`。
- 当前知识源开关关闭时提交 `status = "pending"`、`activeStatus = "disabled"`。

状态设计：

- 笔记列表、当前笔记、标题草稿、正文草稿、目录展开、保存/删除 loading 都放在 `/note` 页面本地 state。
- 笔记编辑状态只服务当前页面，本地 state 能清晰表达页面生命周期。
- `draftRawContent` 是高频输入字段，保留在组件内部可以让 Markdown 编辑器交互更轻量。

### 5.2 知识库卡片展示

卡片展示更能反映真实配置和规模的信息：

- 知识库名称与描述。
- 启用/禁用状态。
- 文档数。
- Chunks 数。
- TopK。
- 标签。

卡片 UI 做了两类改良：

- 状态 badge 使用更大的字号和字重，避免右上角启用/禁用过弱。
- 卡片减少无效留白，统计项字体加粗、字号略增，提升管理后台的信息密度。

标签展示规则：

- 卡片最多展示 5 个标签。
- 标签区域最多两行，不折叠，不显示 `+N`。
- 单个标签在产品层面建议 4 个中文字符以内，技术层面后端限制 8 个字符以内以兼容英文和数字。
- 标签使用圆角矩形、淡色实底、黑色文字。
- 对存量深色标签，前端展示时自动混白变浅，数据库继续保留原始 `color` 值。

### 5.5 文档归属管理

文档归属管理解决的问题是：一个知识库应该引用哪些已经解析完成的知识源文档。

可选文档条件：

```text
DocumentSource.status = "parsed"
DocumentSource.activeStatus = "active"
```

数据流：

```text
进入详情页
-> 请求当前知识库 tree
-> 请求可用知识源文档
-> 根据已绑定 documentId 拆分已引用文档和待选文档
-> 用户启用/撤下文档
-> 前端维护 dirty 状态
-> 点击保存时计算 toAdd/toRemove
-> 调用绑定/解绑接口
-> 保存成功后重新加载详情
```

相关接口：

| 功能             | 接口                                                                  |
| ---------------- | --------------------------------------------------------------------- |
| 获取知识库详情树 | `GET /api/knowledge-bases/[id]/tree`                                  |
| 获取可用知识源   | `GET /api/rag-management/documents?status=parsed&activeStatus=active` |
| 绑定文档         | `POST /api/rag-management/knowledge-bases/[id]/documents`             |
| 解绑文档         | `DELETE /api/rag-management/knowledge-bases/[id]/documents`           |
| 获取文档 chunks  | `GET /api/rag-management/documents/[id]/chunks`                       |

设计理由：启用/撤下操作先只改前端草稿状态，不立即写库。用户可以一次调整多份文档后统一保存，失败时保留当前草稿，避免误操作导致关系频繁写入。

图片插入位：文档管理页，展示已引用文档、待选文档和保存按钮
建议路径：`docs/lzh/images/rag-document-management.png`

### 5.7 RAG Debug 召回测试

Debug tab 的定位是解释一次 RAG 召回，不负责生成最终回答。它和 Agent 对话的区别是：

| 能力                | Agent 对话 | RAG Debug   |
| ------------------- | ---------- | ----------- |
| 生成最终回答        | 是         | 否          |
| 调用真实 RAG 检索   | 是         | 是          |
| 展示 chunk 得分     | 否         | 是          |
| 展示完整 RAG 上下文 | 否         | 是          |
| 保存知识库配置      | 否         | 仅保存 TopK |

Debug 页面结构：

```text
顶部：问题输入框 + Query Rewrite 开关 + Debug 按钮
中部：Query Rewrite 只读展示区
下方左栏：检索模式、TopK、保存配置
下方右栏：召回 chunk 卡片列表、Prompt 预览入口
```

当前版本的实现方式：

- 直接复用 `POST /api/rag/retrieve`，保证 Debug 结果来自真实 RAG 召回链路。
- Debug 结果保存在页面本地，用于本次诊断展示。
- Agent 对话和公共 RAG 检索服务继续共用同一套检索链路。
- Query Rewrite 开关作为前端调试入口展示，底层 rewrite 行为由现有 RAG 服务决定。
- 左侧参数面板聚焦检索模式和 TopK。
- 保存配置时只持久化 `topK`。

Debug 请求：

```ts
{
  query: string;
  mode: "fast" | "balanced" | "detailed";
  scope: {
    knowledgeBaseIds: [currentKnowledgeBaseId];
  }
}
```

结果处理：

```text
retrieve.contexts
-> 按 score 倒序
-> slice(0, topK)
-> 转换为 Debug chunk 卡片
```

Prompt 预览直接展示后端返回的 `llmContext` 和 `references`。当前 TopK 用于控制右侧卡片展示数量，Prompt 预览则保留后端返回的完整上下文，便于观察真实进入 RAG 上下文构建阶段的内容。

图片插入位：RAG Debug 页面，展示输入区、参数栏、召回结果和 Prompt 弹窗
建议路径：`docs/lzh/images/rag-debug.png`

## 6. 状态管理设计

状态设计的核心原则是：跨页面、跨组件共享且需要保持一致的数据进入 Zustand；只服务当前页面交互、变化频繁或关闭页面即失效的状态放在组件本地。

### 6.1 存放在 Zustand 的全局状态

RAG 模块使用 `src/store/slices/knowledge-base-slice.ts`。

| 状态             | 含义                  | 放入 Zustand 的理由                                      |
| ---------------- | --------------------- | -------------------------------------------------------- |
| `items`          | RAG 知识库轻量列表    | 列表页、统计、卡片编辑、删除后都需要同步同一份列表数据。 |
| `selectedId`     | 当前选中的知识库 id   | 详情页、弹窗或跨组件操作需要知道当前上下文。             |
| `selected`       | 当前知识库详情        | 编辑、详情展示和局部更新需要共享同一份详情数据。         |
| `selectedDocs`   | 当前知识库关联文档    | 文档归属、chunk 查看等组件需要读取当前文档集合。         |
| `selectedChunks` | 当前查看的 chunk 集合 | chunk 弹窗和详情定位需要共享。                           |
| `loading`        | RAG 模块加载状态      | 列表初始化和全局操作可以统一反馈加载状态。               |
| `error`          | RAG 模块错误信息      | 请求失败或操作失败时方便统一展示。                       |

这些状态的共同特点是：它们不是某个输入框的临时值，而是当前 RAG 管理模块的业务数据快照。创建、编辑、删除知识库后，直接更新 `items` 和 `selected` 可以避免页面之间出现旧数据。

### 6.2 保持在本地组件 state 的状态

列表页本地状态：

- 搜索输入 `searchInput`。
- 已提交搜索词 `submittedSearchKeyword`。
- 状态筛选 `statusFilter`。
- 排序字段和方向。
- 创建/编辑/删除弹窗开关。
- 表单草稿。
- 标签候选 `allTags`、标签创建弹窗、标签创建错误。

详情页本地状态：

- 当前 tab。
- 文档归属草稿 `selectedDocuments`。
- 初始绑定文档 id，用于判断 dirty。
- 当前搜索词、搜索结果、筛选条件。
- chunk 弹窗开关、当前文档、定位高亮 chunk。
- 保存文档配置的 loading/error。

Debug 本地状态：

- 问题输入。
- Query Rewrite 开关。
- 检索模式。
- TopK 草稿。
- Debug loading/error。
- Debug 返回结果。
- Prompt 弹窗开关。

知识笔记本地状态：

- 笔记目录 `notes`。
- 当前笔记 `activeNote` / `activeNoteId`。
- 标题草稿 `draftTitle`。
- Markdown 正文草稿 `draftRawContent`。
- 标题编辑态、目录展开态。
- 初始化、详情加载、保存、删除 loading。

本地保存的原因：

- 这些状态只影响当前页面或当前弹窗。
- 高频输入和筛选如果进入全局 store，会增加无关组件重渲染。
- 关闭页面或弹窗后不需要保留。
- Debug 结果属于一次性诊断数据，不应该污染全局知识库数据。

### 6.3 状态流转示例

创建或编辑知识库：

```text
打开弹窗
-> 本地 form state 编辑
-> 提交 API
-> 成功后更新 Zustand items
-> 关闭弹窗并刷新标签候选
```

文档归属保存：

```text
详情接口返回已绑定文档
-> 本地保存 initialSelectedDocumentIds
-> 用户启用/撤下文档
-> 本地 selectedDocuments 变化
-> dirty = 当前选择和初始选择不同
-> 保存时计算 toAdd/toRemove
-> 成功后重新加载详情并重置 dirty
```

Debug 测试：

```text
用户输入 query
-> 本地保存 query/mode/topK
-> 调用 /api/rag/retrieve
-> 本地保存 result
-> 右侧渲染 chunk 卡片
-> Prompt 弹窗读取本次 result
```

知识笔记切换：

```text
点击右侧目录项
-> 判断是否正在保存
-> 保存当前笔记草稿
-> 保存成功后加载目标笔记详情
-> 初始化 draftTitle/draftRawContent
-> 保存失败则保留当前笔记，不执行切换
```

## 7. 后端接口与服务设计

### 7.1 知识库服务

主要文件：

- `src/features/knowledge-bases/server/knowledge-base-service.ts`
- `src/features/knowledge-bases/server/schemas.ts`
- `src/features/knowledge-bases/server/mappers.ts`

服务层职责：

- 校验知识库名称、状态、TopK、tagIds 等业务字段。
- 创建、编辑、删除知识库。
- 读取知识库列表和详情。
- include 文档、chunk、tags，并映射为前端 DTO。
- 保存标签绑定关系。
- 清理无人使用的 RAG 标签。

关键设计：

- `tagIds` 可选，旧调用保持原有标签关系。
- 传入 `tagIds: []` 表示清空当前知识库标签。
- 保存标签时后端去重并校验标签是否存在。
- 编辑时使用“全量替换绑定关系”的方式，逻辑简单且避免局部 diff 出错。
- 删除知识库后检查原有关联标签，如果无人使用则删除。

### 7.2 标签接口

主要文件：

- `src/app/api/tags/route.ts`
- `src/server/services/knowledge/tag.service.ts`
- `src/features/knowledge/api/tags.ts`

接口行为：

| 接口                      | 行为                                                  |
| ------------------------- | ----------------------------------------------------- |
| `GET /api/tags`           | 返回全量标签，兼容通用标签管理。                      |
| `GET /api/tags?scope=rag` | 清理无人使用标签后，只返回至少被一个 RAG 使用的标签。 |
| `POST /api/tags`          | 创建标签，校验名称唯一和颜色合法。                    |

设计理由：RAG 编辑弹窗里的候选标签不是全局标签仓库，而是“已经被 RAG 使用过、可复用”的标签集合。这样可以防止用户创建后又取消保存的标签长期堆积在候选栏。

### 7.3 文档与 chunk 接口

主要能力：

- 查询可作为知识源的文档。
- 查询知识库详情树。
- 查询文档下 chunk。
- 绑定或解绑知识库文档。
- 在当前知识库范围内做管理端搜索。

这些接口服务于管理端维护，负责文档归属、chunk 查看和知识内容定位。chunk 查看是只读能力，页面通过弹窗展示原始片段和元信息。

### 7.4 知识笔记服务

主要文件：

- `src/features/note/server/note-service.ts`
- `src/features/note/server/schemas.ts`
- `src/app/api/notes/route.ts`
- `src/app/api/notes/[id]/route.ts`

服务层职责：

- 只读写 `sourceType="markdown"` 且 `fileType="note"` 的 `DocumentSource` 记录。
- 目录列表只返回摘要字段，不返回正文，避免一次性拉取大量 Markdown。
- 详情接口返回 `rawContent`。
- 创建笔记时补齐 `originalName`、`fileName`、`mimeType`、`fileSize` 等文档元信息。
- 更新标题时同步 `title`、`originalName`、`fileName`。
- 更新正文时同步 `rawContent` 和 `fileSize`。
- 删除接口只删除当前笔记记录。

设计理由：笔记本质上也是一种知识源，因此复用 `DocumentSource` 可以让它后续自然进入 RAG 待选文档范围，不需要为笔记再维护一套独立知识源表。

### 7.5 RAG Debug 接口复用

Debug 复用现有后端检索接口：

```text
POST /api/rag/retrieve
```

这样做的好处：

- Debug 看到的是当前真实 RAG 检索链路返回的内容。
- Debug 和 Agent 对话使用同一套检索结果来源，便于对照排查。
- 复用现有接口降低了额外服务维护成本。

当前展示边界：

- 前端 TopK 用于控制 Debug 卡片展示数量。
- Prompt 预览使用后端原始 `llmContext`，可能和右侧展示卡片数量不完全一致。
- Query Rewrite 开关展示调试入口，真实 rewrite 行为由现有 RAG 服务统一处理。

## 8. 关键工程取舍

### 8.1 为什么知识库标签需要关系表

一个标签可以被多个知识库复用，一个知识库也可以有多个标签，因此使用 `KnowledgeBaseTag` 建立多对多关系。相比把标签数组直接存在 `KnowledgeBase` 字符串字段中，关系表更适合：

- 查询某个标签被哪些知识库使用。
- 删除知识库时自动清理绑定。
- 判断标签是否无人使用。
- 后续扩展标签排序、权限或统计。

### 8.2 为什么标签作为管理维度使用

当前标签用于管理端识别和搜索知识库。知识库标签是人工管理维度，chunk 的 `suggestedTags` 是 AI 对知识片段的建议标签，两者粒度不同。分层设计让用户可以清楚区分“知识库展示分类”和“chunk 语义建议”。

### 8.3 为什么 Debug 不做成 Agent 对话

Agent 对话关注“回答是否正确”，Debug 关注“为什么这些 chunk 被召回”。如果 Debug 也生成最终答案，就会和现有 Agent 对话重复。当前设计只展示召回证据、得分、来源和 Prompt 上下文，补齐的是可观测性。

### 8.4 为什么保留 similarityThreshold 字段但隐藏 UI

数据库保留 `similarityThreshold` 兼容旧结构；UI 聚焦当前真实可调的 TopK、状态和标签配置。这个处理让管理端展示内容与实际可调能力保持一致。

### 8.5 为什么不把所有页面状态都放入 Zustand

Zustand 只承担跨组件共享业务数据。搜索框输入、筛选、弹窗开关、表单草稿和 Debug 结果都是页面级临时状态。放入全局 store 会带来三个问题：

- 状态生命周期变长，关闭页面后仍可能残留。
- 高频输入导致更多组件订阅更新。
- 全局 store 语义变混乱，难以区分业务数据和 UI 草稿。

## 9. 兼容性设计

已覆盖的兼容点：

- `KnowledgeBase.similarityThreshold` 作为历史字段继续保留。
- 后端 schema 可继续兼容旧调用中的 `similarityThreshold`。
- `/api/tags` 默认返回全量标签，`scope=rag` 返回 RAG 使用中的标签。
- `/api/notes` 聚焦 Markdown 笔记类型，普通上传文档由文档模块继续维护。
- RAG 标签用于管理端识别和列表搜索，公共 RAG 检索链路保持稳定。
- Debug 结果采用页面本地展示方式，避免引入额外持久化结构。
- 文档解绑通过 `KnowledgeBaseDocument` 调整引用关系，文档本体和 chunk 继续保留。
- chunk 查看保持只读展示，文档归属草稿状态只由启用/撤下操作驱动。

主要边界：

- 新建标签后如果用户取消保存知识库，标签可能短暂存在；下次请求 `scope=rag` 时会被清理。
- Debug 的 TopK 用于前端结果展示数量控制。
- Prompt 预览展示后端真实上下文，右侧卡片展示前端 TopK 截取后的结果。

## 11. 总结

本模块的重点是把 RAG 管理从“配置列表”升级为“可维护、可解释、可调试的知识库工作台”。知识库标签解决管理识别问题，文档归属解决知识源复用问题，chunk 搜索和定位解决内容维护问题，Debug 解决召回可观测性问题。

从工程实现上看，模块形成了完整闭环：

- 数据库有真实关系表和业务规则。
- 后端有服务层校验、写入、查询和清理逻辑。
- 前端有列表、详情、标签编辑、文档归属和 Debug 视图。
- Zustand 只保存跨组件业务数据，本地 state 承载页面临时交互。
- Debug 复用真实 RAG 接口，避免和 Agent 对话链路割裂。

答辩时可以围绕三条主线展开：一是知识库标签和文档归属如何形成可维护的数据结构；二是 Debug 如何补齐 RAG 召回可解释性；三是全局状态和本地状态如何划分，保证页面复杂但状态边界清晰。
