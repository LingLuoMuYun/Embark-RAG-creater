# 成员 A 技术文档：知识生命周期管理与后台布局

## 1. 实现功能概览

本人负责分工文档中的成员 A 部分，即知识生命周期管理，同时兼顾项目后台整体布局。具体实现内容包括：

- 管理后台整体布局：实现统一的 Header、Sidebar、Main Content 后台框架，统一承载知识库、知识笔记、知识文档、专家 Agent、专家对话等页面。
- 导航与路由承载：实现 `/` 默认跳转、Sidebar 一级导航、知识库父子导航、当前页面标题解析和路由选中态匹配。
- 知识库生命周期管理：实现知识库列表、搜索、排序、状态筛选、统计卡片、新建、编辑、删除和状态展示。
- 知识库详情页：实现从知识库列表进入详情页，在后台布局内查看知识库基础信息。
- 文档归属管理：实现已引用文档、待选知识源文档、启用/撤下、保存文档配置和保存失败回滚展示。
- Chunk 明细展示：在知识库详情页通过弹窗查看已引用文档和待选知识源文档的知识分片，只读展示 chunk 内容和元信息。
- 知识笔记管理：实现 Markdown 笔记创建、目录加载、详情加载、标题编辑、正文编辑与预览、保存、切换前自动保存、删除和右侧目录收缩展开。
- 知识源状态控制：在知识笔记中实现“是否成为知识源”开关，控制笔记是否可被 RAG 知识库引用。
- 服务端接口与校验：为知识库、文档归属、文档 chunk、知识笔记提供 API Route、Zod 校验、服务层和统一响应处理。

本部分对应分工文档中“知识生命周期管理”的核心目标：知识从创建、维护、状态控制、归属配置到被知识库引用，形成可管理、可追踪、可复用的数据基础。

## 2. 技术选型

- 前端框架：Next.js 16 App Router、React、TypeScript。
- 样式方案：Tailwind CSS，组件风格沿用 shadcn/ui。
- UI 组件：`Button`、`Input`、`Dialog`、`AlertDialog`、`Select`、`Switch`、`Card`、`Badge` 等通用组件。
- 图标库：`lucide-react`。
- 状态管理：Zustand 管理知识库轻量列表和当前选中详情；页面临时状态使用组件本地 state。
- 表单与接口校验：Zod。
- 数据库：SQLite。
- ORM：Prisma 7，使用 `@prisma/adapter-libsql`，通过 `src/lib/db.ts` 统一访问 Prisma Client。
- Markdown 编辑器：`@uiw/react-md-editor`。

## 3. 后台布局实现

布局相关代码位于：

- `src/components/layout/admin-shell.tsx`
- `src/components/layout/admin-header.tsx`
- `src/components/layout/admin-sidebar.tsx`
- `src/components/layout/admin-nav.ts`

整体结构：

```text
AdminShell
├── AdminSidebar
└── Main Area
    ├── AdminHeader
    └── Main Content
```

实现方式：

- `AdminShell` 作为后台统一容器，负责组合 Sidebar、Header 和业务页面内容。
- `AdminShell` 使用 `usePathname()` 获取当前路由，并通过导航配置解析页面标题和选中状态。
- `AdminSidebar` 根据 `adminNavItems` 渲染一级导航和知识库子导航。
- `AdminHeader` 展示当前页面标题，并提供 Sidebar 展开/收缩按钮和用户信息占位。
- Sidebar 展开状态只保存在 `AdminShell` 本地 state，不进入全局 store。

路由规则：

- `/` 默认重定向到 `/knowledge-bases`。
- `/knowledge-bases` 和 `/knowledge-bases/[id]` 归属 `知识库管理`。
- `/note` 归属 `知识笔记`。
- `/documents` 归属 `知识文档`。
- `/agents` 归属 `专家 Agent`。
- `/agents/chat` 作为 `专家对话` 一级导航，使用精确匹配避免被 `/agents` 误选中。

## 4. 数据模型设计

分工文档和 `docs/DataType.md` 中将 A 部分描述为知识生命周期管理。当前代码中，该部分实际落地时使用以下 Prisma 模型承载：

- `KnowledgeBase`：知识库基础信息和检索配置。
- `DocumentSource`：统一文档来源模型，承载上传文档、Markdown 笔记和其他可作为知识源的文档。
- `KnowledgeBaseDocument`：知识库与文档的多对多引用关系。
- `DocumentChunk`：文档切分后的知识分片。

说明：

- `docs/DataType.md` 中的 `KnowledgeDocument` 职责，在当前 schema 中对应 `DocumentSource`。
- `docs/DataType.md` 中的 `KnowledgeChunk` 职责，在当前 schema 中对应 `DocumentChunk`。
- 这样设计后，外部上传文档和用户 Markdown 笔记都可以统一作为知识源，被知识库引用。

核心关系：

```text
KnowledgeBase
  └── KnowledgeBaseDocument
        └── DocumentSource
              └── DocumentChunk
```

关键字段：

- `KnowledgeBase.status`：控制知识库启用或禁用。
- `KnowledgeBase.topK`：控制检索返回数量。
- `KnowledgeBase.similarityThreshold`：控制相似度阈值。
- `DocumentSource.fileType`：区分外部文档和知识笔记，例如 `file`、`note`。
- `DocumentSource.sourceType`：区分来源类型，例如 `manual`、`file`、`markdown`。
- `DocumentSource.status`：控制文档处理或知识源状态，例如 `pending`、`parsed`。
- `DocumentSource.activeStatus`：控制业务启用状态。
- `DocumentSource.rawContent`：保存文档解析文本或 Markdown 笔记正文。
- `DocumentSource.chunkCount`：冗余保存 chunk 数量，便于列表展示。
- `DocumentChunk.chunkType`：区分普通文本分片和 AI 提取知识分片。

## 5. 知识库生命周期管理

页面入口：

- `src/app/knowledge-bases/page.tsx`
- `src/features/knowledge-bases/index.tsx`

功能实现：

- 初始化加载知识库列表。
- 展示知识库卡片列表。
- 支持按关键字搜索知识库名称和描述。
- 支持按更新时间、文档数量排序。
- 支持排序方向切换。
- 支持按全部、启用、禁用状态筛选。
- 展示知识库总量、启用数量、禁用数量统计卡片。
- 支持新建知识库。
- 支持编辑知识库名称、描述、图标、检索参数和状态。
- 支持删除知识库并进行二次确认。

状态设计：

- 知识库轻量列表存入 Zustand 的 `items`。
- 当前选中知识库 id 存入 `selectedId`。
- 当前知识库详情存入 `selected`。
- 当前知识库文档列表存入 `selectedDocs`。
- 当前查看的分片存入 `selectedChunks`。
- 搜索、排序、筛选、弹窗、表单草稿等 UI 状态保存在页面本地。

这样可以避免把文档和分片明细写入主列表，防止列表状态过重，也方便详情页按需加载。

## 6. 知识库详情与文档归属管理

页面入口：

- `src/app/knowledge-bases/[id]/page.tsx`
- `src/features/knowledge-bases/components/knowledge-base-detail-feature.tsx`

实现目标：

- 点击知识库卡片进入 `/knowledge-bases/[id]`。
- 详情页仍然处于后台 `AdminShell` 内。
- 展示当前知识库基础信息。
- 展示已引用文档列表。
- 展示待选知识源文档列表。
- 支持把待选文档“启用”到当前知识库。
- 支持把已引用文档从当前知识库“撤下”。
- 支持手动保存文档配置。
- 支持通过弹窗查看已引用文档和待选文档的 chunk 明细。

文档可选条件：

```text
DocumentSource.activeStatus = "active"
DocumentSource.status = "parsed"
```

实现方式：

- 页面初始化时请求知识库 tree，获取知识库基础信息、已引用文档和 chunks。
- 同时请求可作为知识源的 `DocumentSource` 列表。
- 前端根据已引用文档 id 过滤待选文档。
- 已引用文档的 chunk 数据来自知识库 tree，待选文档的 chunk 数据在点击“分片”时通过 `GET /api/rag-management/documents/[id]/chunks` 按需加载。
- 点击“分片”只打开只读弹窗，不改变文档归属关系，也不触发 `dirty` 状态。
- 点击“启用”或“撤下”只更新前端编排状态，不立即请求后端。
- 页面维护 `dirty` 状态，表示文档归属关系存在未保存变更。
- 点击“保存文档配置”后计算 `toAdd` 和 `toRemove`。
- `toAdd` 调用绑定接口写入 `KnowledgeBaseDocument`。
- `toRemove` 调用解绑接口删除 `KnowledgeBaseDocument`。
- 保存成功后重新加载详情数据。
- 保存失败时保留当前前端编排状态，并提示错误。

## 7. Chunk 明细展示

Chunk 展示组件位于：

- `src/features/knowledge-bases/components/document-chunk-list.tsx`
- `src/features/knowledge-bases/components/document-chunks-dialog.tsx`

实现方式：

- `document-chunks-dialog.tsx` 作为统一弹窗承载 chunk 明细，已引用文档和待选文档共用同一套查看入口。
- 已引用文档优先复用知识库 tree 中已返回的 chunk 数据，避免重复请求。
- 待选文档未出现在当前知识库 tree 中，点击“分片”时按文档 id 请求 chunk 列表。
- 分片明细不再以内联展开形式展示，避免文档列表被长内容撑开。
- 弹窗支持加载、错误、空状态和搜索结果定位高亮。

展示内容：

- 分片序号。
- 分片类型。
- 分片正文。
- 字符范围。
- 标题、建议分类、建议标签、审核状态等可选字段。

边界限制：

- 当前页面只读展示 chunk。
- 不在详情页新增 chunk。
- 不在详情页编辑 chunk。
- 不在详情页删除 chunk。
- 不在详情页触发重新切分或 embedding。
- 查看待选文档 chunk 不等于启用文档，不会写入 `KnowledgeBaseDocument`。

## 8. 知识笔记管理

页面入口：

- `src/app/note/page.tsx`
- `src/features/note/index.tsx`

组件拆分：

- `note-topbar.tsx`：顶部标题、更新时间、保存、新建、删除和知识源开关。
- `note-editor.tsx`：Markdown 编辑器。
- `note-directory.tsx`：右侧文档目录。
- `note-empty-state.tsx`：空状态。
- `delete-note-dialog.tsx`：删除确认框。

功能实现：

- 进入 `/note` 后请求笔记目录。
- 目录接口只返回摘要，不返回 `rawContent`。
- 如果存在笔记，默认选中更新时间最新的一篇。
- 选中后请求详情接口加载 `rawContent`。
- 支持 Markdown 正文编辑与实时预览。
- 支持点击标题进入编辑态。
- 标题通过 `onBlur` 或 Enter 保存。
- 支持保存当前笔记。
- 点击目录切换文档前自动保存当前笔记。
- 保存失败时不切换文档，保留当前草稿。
- 支持新建笔记。
- 支持删除笔记并二次确认。
- 右侧目录支持收缩和展开。
- 没有笔记时展示空状态和创建入口。

知识笔记数据落库：

- 复用 `DocumentSource`。
- `fileType = "note"`。
- `sourceType = "markdown"`。
- Markdown 正文保存到 `rawContent`。
- `fileSize` 根据 `rawContent` 的 UTF-8 字节数计算。

知识源开关：

- 打开开关时，将当前笔记 `status` 更新为 `parsed`。
- 关闭开关时，将当前笔记 `status` 更新为 `pending`。
- 该开关只改变笔记是否可作为 RAG 待选知识源，不生成 chunk，不绑定知识库，不修改正文。

## 9. API 与服务层实现

知识库相关 API：

- `GET /api/rag-management/knowledge-bases`
- `POST /api/rag-management/knowledge-bases`
- `PATCH /api/rag-management/knowledge-bases/[id]`
- `DELETE /api/rag-management/knowledge-bases/[id]`
- `GET /api/rag-management/knowledge-bases/[id]/tree`
- `POST /api/rag-management/knowledge-bases/[id]/documents`
- `DELETE /api/rag-management/knowledge-bases/[id]/documents`
- `GET /api/rag-management/documents`
- `GET /api/rag-management/documents/[id]/chunks`

知识笔记相关 API：

- `GET /api/notes`
- `POST /api/notes`
- `GET /api/notes/[id]`
- `PATCH /api/notes/[id]`
- `DELETE /api/notes/[id]`

服务层文件：

- `src/features/knowledge-bases/server/knowledge-base-service.ts`
- `src/features/knowledge-bases/server/knowledge-chunk-service.ts`
- `src/features/knowledge-bases/server/knowledge-search-service.ts`
- `src/features/knowledge-bases/server/mappers.ts`
- `src/features/knowledge-bases/server/schemas.ts`
- `src/features/note/server/note-service.ts`
- `src/features/note/server/schemas.ts`

实现规则：

- Route Handler 只负责解析请求、校验入参、调用服务和返回响应。
- Zod schema 负责校验请求体和路由参数。
- Service 负责业务规则和 Prisma 操作。
- Mapper 负责 Prisma 数据结构到前端 DTO 的转换。
- 多表写入使用 Prisma transaction。
- 统一通过 `src/lib/api-response.ts` 返回成功或错误响应。

## 10. 前端数据流与状态管理

知识库列表数据流：

```text
/knowledge-bases 页面加载
-> 请求知识库列表
-> 归一化后写入 Zustand items
-> 页面从 items 派生搜索、筛选、排序和统计结果
-> 渲染知识库卡片
```

知识库详情数据流：

```text
进入 /knowledge-bases/[id]
-> 请求 knowledgeBase tree
-> 请求可用知识源文档
-> 拆分已引用文档和待选文档
-> 点击分片时打开弹窗；待选文档按需请求 chunks，已引用文档复用 tree chunks
-> 用户启用/撤下文档
-> 计算 dirty
-> 保存时提交 toAdd/toRemove
-> 成功后重新加载详情
```

知识笔记数据流：

```text
进入 /note
-> 请求笔记目录
-> 默认选中最新笔记
-> 请求笔记详情
-> 编辑标题或 Markdown 正文
-> 保存到 DocumentSource
-> 切换文档前自动保存当前草稿
```

状态边界：

- 知识库列表和当前知识库详情使用 Zustand。
- 知识笔记草稿正文使用页面本地 state。
- 文档归属编排状态使用详情页本地 state。
- Sidebar 展开状态使用布局本地 state。

## 11. 错误处理与边界控制

已处理的关键边界：

- `/agents/chat` 不会被 `/agents` 错误匹配。
- `/knowledge-bases/[id]` 不新增 Sidebar 导航项，仍归属知识库管理。
- 知识库删除不删除文档和 chunk，只删除知识库及关联关系。
- 文档解绑只删除 `KnowledgeBaseDocument`，不删除 `DocumentSource` 和 `DocumentChunk`。
- 查看 chunk 不改变文档归属关系，不会触发 `dirty` 状态。
- 待选文档查看 chunk 只是只读预览，不等于启用到当前知识库。
- 知识笔记接口只操作 `fileType = "note"` 且 `sourceType = "markdown"` 的记录。
- 笔记切换前保存失败时不切换，避免草稿丢失。
- RAG 详情页保存文档配置失败时保留前端编排状态。
- 待选知识源必须满足 `activeStatus = "active"` 和 `status = "parsed"`。
- API 入参使用 Zod 校验，不依赖前端类型。
