# Prisma Schema 合并变更分析

> 变更日期：2026-06-02
> 变更范围：Prisma Schema + 服务层 + API 路由 + 前端组件

---

## 一、变更概述

本次重构将 4 个独立的 Prisma 模型合并为 2 个，消除了三条平行管线的数据孤岛。

### 删除的模型

| 模型 | 原因 |
|---|---|
| `KnowledgeDocument` | 字段与 `DocumentSource` 高度重叠，合并到 `DocumentSource` |
| `KnowledgeChunk` | 字段是 `DocumentChunk` 的子集，合并到 `DocumentChunk` |
| `CandidateKnowledge` | AI 提炼结果死端，合并到 `DocumentChunk`（`chunkType="knowledge"`） |

### 变更前后对比

```
变更前 (6 模型)                          变更后 (3 模型)
═══════════════                          ═══════════════
DocumentSource ──→ DocumentChunk         DocumentSource ──→ DocumentChunk
                                       (统一文档模型)     (统一分块模型)
KnowledgeDocument ──→ KnowledgeChunk          │              ├─ type="text"
                                              │              └─ type="knowledge"
CandidateKnowledge (孤岛)
```

## 二、各模型变更详情

### 2.1 DocumentSource（合并 KnowledgeDocument）

```diff
+ title              String         // ← KnowledgeDocument.title
+ sourceType         String         // ← KnowledgeDocument.sourceType (manual|file|url|text|markdown|image)
+ fileName           String?        // ← KnowledgeDocument.fileName
+ fileUrl            String?        // ← KnowledgeDocument.fileUrl
+ mimeType           String?        // ← KnowledgeDocument.mimeType
+ rawContent         String?        // 重命名自 content
+ chunkSize          Int            // ← KnowledgeDocument.chunkSize (默认 800)
+ chunkOverlap       Int            // ← KnowledgeDocument.chunkOverlap (默认 100)
+ activeStatus       String         // ← KnowledgeDocument.status (active|disabled)
+ error              String?        // 统一自 errorMessage
+ knowledgeBaseId    String?        // ← KnowledgeDocument.knowledgeBaseId (遗留兼容)
- content            (移除)         // → rawContent
- errorMessage       (移除)         // → error
- candidateKnowledge (移除)         // → DocumentChunk where chunkType="knowledge"
```

**状态字段双轨设计：**
| 字段 | 语义 | 取值 |
|---|---|---|
| `status` | 处理流水线状态 | uploading → uploaded → pending → parsing → parsed → failed |
| `activeStatus` | 业务启用状态 | active \| disabled |

### 2.2 DocumentChunk（合并 KnowledgeChunk + CandidateKnowledge）

```diff
+ documentSourceId   String?        // 改为可选（支持无来源文档的文本提炼）
+ chunkType          String         // "text"(机械分块) | "knowledge"(AI提炼)
+ title              String?        // ← CandidateKnowledge.title
+ suggestedCategory  String?        // ← CandidateKnowledge.suggestedCategory
+ suggestedTags      String?        // ← CandidateKnowledge.suggestedTags (JSON)
+ reviewStatus       String?        // ← CandidateKnowledge.status (pending|confirmed|rejected)
  embedding          String?        // (已有，来自 KnowledgeChunk)
  chunkStatus        String         // (已有) active|disabled
```

**两种 chunk 类型对比：**

| 维度 | chunkType="text" | chunkType="knowledge" |
|---|---|---|
| 来源 | 文件解析 → 机械/语义分块 | AI LLM 提炼 |
| title | null | 知识点标题 |
| suggestedCategory | null | AI 建议分类 |
| suggestedTags | null | AI 建议标签 (JSON) |
| reviewStatus | null | pending → confirmed/rejected |
| chunkStatus | "active" | "disabled" → "active" (确认后) |
| embedding | 创建时生成 | 确认时生成 |

**完整字段矩阵：**

| 字段 | text 块 | knowledge 块 | 继承自 |
|---|---|---|---|
| `id` | ● | ● | 两者共有 |
| `documentSourceId` | ● | ○ (可选) | DocumentChunk |
| `knowledgeBaseId` | ● | ○ | KnowledgeChunk |
| `content` | ● | ● | 两者共有 |
| `chunkIndex` | ● | ● | 两者共有 |
| `charStart` / `charEnd` | ● | ● | DocumentChunk |
| `embedding` | ● | ● | KnowledgeChunk |
| `chunkType` | "text" | "knowledge" | **新增** |
| `title` | — | ● | CandidateKnowledge |
| `suggestedCategory` | — | ● | CandidateKnowledge |
| `suggestedTags` | — | ● | CandidateKnowledge |
| `reviewStatus` | — | ● | CandidateKnowledge |
| `chunkStatus` | "active" | pending→active | KnowledgeChunk |

## 三、合并后的管线变化

### 3.1 文档导入 + RAG 检索（统一管线）

```
文件上传 → DocumentSource（status: uploading → parsed）
  ├─ 机械分块 → DocumentChunk (type="text", status="active")
  │              └─ embedding 生成 → ChunkEmbedding
  │                  └─ RAG 可检索 ✓
  │
  └─ AI 提炼 → DocumentChunk (type="knowledge", status="disabled", reviewStatus="pending")
                 └─ 审核确认 → status="active", reviewStatus="confirmed"
                     └─ embedding 生成 → ChunkEmbedding
                         └─ RAG 可检索 ✓
```

**关键改进：** 确认后的知识块自然进入 RAG 检索管线，不再存在"确认即死端"的问题。

### 3.2 RAG 检索中的 chunkType 映射

检索时 `chunk-repository.ts` 将 `chunkType` 映射为 RAG 域类型：

| DB chunkType | RAG chunkType | 说明 |
|---|---|---|
| `"text"` | `"text"` | 原始文档分块 |
| `"knowledge"` | `"summary"` | AI 提炼的知识（可在 RAG scope 中单独筛选） |

### 3.3 审核工作流

```
AI 提炼 → pending → 用户编辑 → 确认 (自动 embedding) → RAG 可检索
                  → 拒绝 (chunkStatus 保持 disabled)
```

## 四、优点分析

### 4.1 架构层面

| 优点 | 说明 |
|---|---|
| **消除数据孤岛** | 文档、分块、AI 知识统一存储，不再有三套独立的表结构 |
| **消除死端管线** | AI 提炼的知识确认后自动进入 RAG 检索，闭环完成 |
| **单表多态** | 用 `chunkType` 区分分块类型，避免多表 JOIN |
| **减少表数量** | 6 表 → 3 表，关系复杂度大幅降低 |
| **复用 embedding 基础设施** | text 分块和 knowledge 分块共用同一套 `ChunkEmbedding` / `indexChunks()` 管线 |

### 4.2 开发层面

| 优点 | 说明 |
|---|---|
| **统一 CRUD** | 一套 API/服务层即可操作所有分块类型，不用维护三套接口 |
| **审核即入库** | `confirmCandidates()` 一次调用完成状态更新 + embedding 生成 |
| **前端兼容** | API 端点 URL 保持不变，前端组件几乎无需修改 |

### 4.3 数据层面

| 优点 | 说明 |
|---|---|
| **检索统一** | RAG 检索一次查询即可覆盖原始分块 + AI 提炼知识 |
| **灵活筛选** | 可通过 `chunkType` 和 `reviewStatus` 精确控制检索范围 |

## 五、问题与风险分析

### 5.1 当前问题

| # | 问题 | 严重程度 | 说明 |
|---|---|---|---|
| 1 | **status 字段语义过载** | 🟡 中 | DocumentSource 有 `status`(6 个处理状态) + `activeStatus`(启用/禁用) + DocumentChunk 有 `chunkStatus`(启用/禁用) + `reviewStatus`(审核状态)，容易混淆 |
| 2 | **documentSourceId 可空** | 🟡 中 | 从文本直接提炼的知识块无来源文档，`documentSourceId` 为 null，无法追溯来源 |
| 3 | **chunkType 子类型丢失** | 🟢 低 | CandidateKnowledge 的 `type` 枚举(faq/concept/procedure/note/summary)在合并后通过 `chunkType="knowledge"` 统一表示，子类型信息未持久化到独立字段 |
| 4 | **KnowledgeCategory 未关联** | 🟢 低 | `suggestedCategory` 是纯文本字符串，未与 `KnowledgeCategory` 表建立外键关联 |
| 5 | **tags 为 JSON 字符串** | 🟢 低 | `suggestedTags` 以 JSON 字符串存储，未与 `KnowledgeTag` 表建立多对多关系 |

### 5.2 潜在风险

| # | 风险 | 说明 |
|---|---|---|
| 1 | **单表膨胀** | DocumentChunk 承载了 3 种模型的全部字段（15 个字段），未来继续扩展会加剧膨胀 |
| 2 | **NULL 字段泛滥** | text 块永远不需要 `title`/`suggestedCategory`/`suggestedTags`/`reviewStatus`；knowledge 块的 `charStart`/`charEnd` 意义不大 |
| 3 | **Prisma 类型复杂** | TypeScript 类型不能根据 `chunkType` 做收窄，业务代码需手动判断 null |
| 4 | **从文本提炼的 knowledge 块** | 无 `documentSourceId`、无 `knowledgeBaseId`，在 RAG 检索中会被过滤掉（`knowledgeBaseId` 为 null 时跳过），实际上仍不可检索 |
| 5 | **迁移不可逆** | 已删除 3 个模型和旧数据库，无法回滚到旧架构 |

### 5.3 改进建议

| 建议 | 优先级 | 说明 |
|---|---|---|
| 用 Prisma discriminator 或视图区分 chunkType | 🟡 中 | 若 Prisma 后续支持多态，可更优雅地处理两种 chunk 的字段差异 |
| 为 knowledge 块自动关联 knowledgeBaseId | 🟡 中 | 从文档提炼时，应继承来源文档的 knowledgeBaseId |
| 建立 Category/Tag 外键关联 | 🟢 低 | 将 `suggestedCategory` / `suggestedTags` 改为真实的外键关系 |
| 为 knowledge 块保留子类型字段 | 🟢 低 | 增加 `knowledgeType` 字段存储 faq/concept/procedure/note/summary |
| 统一命名规范 | 🟢 低 | `status`/`activeStatus`/`chunkStatus`/`reviewStatus` 建议统一为 `pipelineStatus`/`enabled`/`reviewStatus` |

## 七、最终模型关系图

```
KnowledgeBase
  │ 1:N
  ▼
KnowledgeBaseDocument (连接表，含 sortOrder)
  │ N:1
  ▼
DocumentSource (统一文档)
  │ 1:N
  ├──→ DocumentChunk (chunkType="text")     ──→ ChunkEmbedding
  │     原始机械/语义分块，创建时即生成 embedding
  │
  └──→ DocumentChunk (chunkType="knowledge") ──→ ChunkEmbedding
        AI 提炼知识，审核确认后生成 embedding
        reviewStatus: pending → confirmed/rejected
        chunkStatus:   disabled → active
```
