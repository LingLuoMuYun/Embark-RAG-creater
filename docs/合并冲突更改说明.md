# 文档管理模块合并变更文档

> 合并日期：2026-06-01  
> 参考项目：`rag-creater-zsh`  
> 目标项目：`rag-creater-A-rag-manager-lzh`  
> 合并重点：文档管理部分（documents 文档管理模块）

---

## 一、合并概览

将参考项目完整的「上传 → 解析 → 分段 → AI 提取 → 候选审核」文档管理流水线合并到当前项目，同时保留当前项目的知识库 M:N 关联结构和现有功能。

### 核心原则

- **数据库**：保留当前项目数据库内容为基础，新增参考项目的模型
- **路由冲突**：`/api/documents` 替换为 DocumentSource 流水线，旧 KnowledgeDocument CRUD 迁移至 `/api/knowledge-documents`
- **不做额外重构**：保持改动聚焦，不删除已有功能

## 二、数据库变更

### 2.1 Prisma Schema（`prisma/schema.prisma`）

**保留的 4 个模型**（结构不变）：

| 模型 | 说明 |
|------|------|
| `KnowledgeBase` | RAG 知识库（M:N 关联文档） |
| `KnowledgeDocument` | 正式知识文档 |
| `KnowledgeBaseDocument` | M:N 连接表 |
| `KnowledgeChunk` | 知识文档切块 |

**新增的 3 个模型**：

| 模型 | 说明 |
|------|------|
| `DocumentSource` | 上传解析工作区（文件上传入口，状态流转：uploading → uploaded → parsing → parsed/failed） |
| `DocumentChunk` | DocumentSource 的文本切块（chunkIndex、content、charStart、charEnd） |
| `CandidateKnowledge` | AI 提取候选知识（title、content、suggestedCategory、suggestedTags、type、status） |

### 2.2 数据架构关系

```
DocumentSource ──< DocumentChunk       (上传→解析→分段 流水线)
     └── CandidateKnowledge           (AI 提取→审核)

KnowledgeBase ──< KnowledgeBaseDocument >── KnowledgeDocument
     └── KnowledgeChunk               (正式知识库，M:N 关系保持不变)
```

## 七、API 路由变更（共 20 个路由文件）

### 7.1 冲突处理详情

> **关键冲突**：`/api/documents` 路径在当前项目和参考项目中有完全不同的实现。
>
> - 当前项目：操作 `KnowledgeDocument` 模型（正式知识库文档）
> - 参考项目：操作 `DocumentSource` 模型（文档上传流水线）
>
> **解决方案**：`/api/documents` → 替换为 DocumentSource 流水线；旧 KnowledgeDocument CRUD → 迁移到 `/api/knowledge-documents`

### 7.2 替换的路由（3 个）

| 路由 | 方法 | 说明 |
|------|------|------|
| `src/app/api/documents/route.ts` | GET/POST | **替换**：KnowledgeDocument CRUD → DocumentSource 列表 + 文件上传（FormData） |
| `src/app/api/documents/[id]/route.ts` | GET/PATCH/DELETE | **替换**：KnowledgeDocument 操作 → DocumentSource 详情/更新/删除 |
| `src/app/api/documents/[id]/chunks/route.ts` | GET | **替换**：KnowledgeChunk 查询 → DocumentChunk 查询 |

## 十一、功能流水线

完整的文档管理流程：

```text
上传文件（11 种格式，最大 100MB）
  → 解析文档（编码检测 → 文本提取）
  → 文本分段（语义向量切分 → 机械切分回退）
  → AI 提取（逐块 LLM 调用 → 5 层 JSON 容错 → 跨块去重）
  → 候选审核（编辑/确认/拒绝/重试）
  → 沉淀到知识库
```
