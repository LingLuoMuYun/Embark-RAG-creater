## 一、模块概览

### 1.0 业务背景

在企业知识库管理系统中，文档导入只是第一步。解析后的分段是"原材料"，还需要经过 AI 提炼、人工审核、向量化等环节才能变成可检索的"知识"。

本模块解决的核心问题是：**将文档分段通过 LLM 自动提炼为结构化知识条目，并提供审核工作台让人工把关质量，最终将确认的知识送入 RAG 检索管线**。这一环节是知识从"原始文档"到"可检索知识"的关键转化工序。

### 1.1 在系统中的位置

**一句话职责**：将文档分段通过 AI 提炼为结构化知识，经人工审核后入库，打通从文档到 RAG 检索的最后一公里。

```
成员C(文档导入管线)
    → 文件上传 → 文本解析 → 语义分段
    → 成员D(AI知识提炼) ← 本模块
        → LLM 提炼 → 候选知识 → 审核确认 → 向量化 → 知识入库
    → 成员A(知识生命周期管理)
    → RAG 检索管线
```

### 1.2 解决的核心问题

| 问题 | 方案 |
|------|------|
| 文档内容无法直接编辑 | 提供全文编辑 + 自动重分段能力 |
| AI 提炼的知识质量不可控 | 审核工作台：人工编辑 + 确认入库 |
| 提炼结果与 RAG 检索断裂 | 确认后自动生成 embedding，进入检索管线 |
| 原有三套数据模型形成孤岛 | 重构 Prisma Schema，统一为单一 DocumentChunk 模型 |

### 1.3 关键指标

| 指标 | 数值 |
|------|------|
| API 接口 | 12 个（全部 Zod 校验） |
| 知识类型 | 5 种（faq/concept/procedure/note/summary） |
| JSON 解析容错 | 5 层策略 |
| LLM 调用重试 | 2 次（500ms 间隔） |
| 去重策略 | title 相似度匹配 |

## 二、技术架构

### 2.1 AI 知识提炼流程

```
文档已解析(status=parsed)
    → POST /api/ai/extract/from-document { documentId }
    → extractFromDocument() 核心编排:
        1. 校验文档状态（必须为 parsed）
        2. 获取文档分段（chunks where chunkType="text"）
        3. 逐段调用 LLM 提炼:
            → renderChunkUserPrompt() 构造上下文（附加文档名+段落位置）
            → extractKnowledge() 调用 LLM + 5 层 JSON 容错
            → CandidateKnowledgeSchema Zod 校验
        4. 合并所有候选知识
        5. deduplicateCandidates() 按 title 相似度去重
        6. 写入 DocumentChunk(chunkType="knowledge", reviewStatus="pending")
    → 返回提炼结果

审核确认流程:
    → POST /api/knowledge/candidates/confirm { ids, knowledgeBaseIds }
    → confirmCandidates():
        1. 查询 pending 状态的 knowledge chunk
        2. 确保文档已绑定到目标知识库（不存在则创建关联）
        3. indexChunks() 生成向量索引
           → 失败则回滚：删除 embedding + 恢复 disabled 状态
        4. 更新状态: reviewStatus="confirmed", chunkStatus="active"
        5. 知识进入 RAG 可检索状态
```

### 2.2 数据模型

**DocumentChunk — 统一分段模型（text + knowledge 共用）**

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String | 主键 |
| `documentSourceId` | String? | 关联文档（knowledge 块可为空，支持文本直接提炼） |
| `content` | String | 分段文本/知识内容 |
| `chunkIndex` | Int | 分段序号 |
| `charStart` / `charEnd` | Int | 原文字符偏移量（可追溯） |
| `chunkType` | String | `"text"`（机械/语义分段）或 `"knowledge"`（AI 提炼） |
| `title` | String? | 知识标题（仅 knowledge 类型） |
| `suggestedCategory` | String? | AI 建议分类 |
| `suggestedTags` | String? | AI 建议标签（JSON 数组） |
| `knowledgeType` | String? | 知识子类型：faq/concept/procedure/note/summary |
| `reviewStatus` | String? | 审核状态：pending/confirmed/rejected |
| `chunkStatus` | String | 启用状态：active/disabled |
| `embedding` | String? | 向量表示（确认后生成） |
| `knowledgeBaseId` | String? | 归属知识库 ID（确认后写入） |

**两种 chunk 类型对比：**

| 维度 | chunkType="text" | chunkType="knowledge" |
|------|------------------|----------------------|
| 来源 | 文件解析 → 机械/语义分块 | AI LLM 提炼 |
| title | null | 知识点标题 |
| suggestedCategory | null | AI 建议分类 |
| suggestedTags | null | AI 建议标签 (JSON) |
| knowledgeType | null | faq/concept/procedure/note/summary |
| reviewStatus | null | pending → confirmed/rejected |
| chunkStatus | "active" | disabled → active (确认后) |
| embedding | 创建时生成 | 确认时生成 |

## 三、核心能力

### 3.1 AI 知识提炼引擎

**文件**：`src/lib/ai-extract.ts`

#### 3.1.1 知识类型定义

| 类型 | 说明 | 示例 |
|------|------|------|
| `faq` | 问答对 | "如何配置数据库？" → 步骤说明 |
| `concept` | 概念解释 | "什么是微服务架构" → 定义和说明 |
| `procedure` | 操作步骤 | "部署流程" → 1. 2. 3. 步骤 |
| `note` | 注意事项 | "生产环境注意事项" → 提醒列表 |
| `summary` | 总结摘要 | 对一段内容的概括 |

**为什么选择这 5 种类型？**
- 覆盖企业知识的主要形态：概念解释、操作步骤、注意事项
- faq 是最常见的知识形式，适合问答场景
- summary 用于提炼长文档的核心观点
- 这 5 种类型与 RAG 检索的使用场景天然匹配

#### 3.1.2 LLM 调用流程

```typescript
// 核心提炼函数
async function extractKnowledge(text: string): Promise<ExtractResult> {
  for (let attempt = 0; attempt <= 2; attempt++) {
    try {
      const rawResult = await callLLM(text);
      const parsed = CandidateKnowledgeSchema.parse(rawResult);
      return { success: true, candidates: parsed };
    } catch (err) {
      if (attempt === 2) return { success: false, error: message, retryable: true };
      await sleep(500); // 500ms 后重试
    }
  }
}
```

#### 3.1.3 JSON 解析容错（5 层策略）

**为什么需要 5 层容错？**

LLM 输出 JSON 的常见问题：
1. **尾部多余内容**：模型在 JSON 后附加解释性文字或 markdown 标记
2. **尾部逗号**：JavaScript 风格的非标准 JSON
3. **无引号的 key**：非标准 JSON 格式
4. **截断输出**：token 限制导致 JSON 不完整
5. **代码块包裹**：```json ... ``` 格式

单一修复策略无法覆盖所有场景，因此设计了 5 层递进式容错：

| 层级 | 策略 | 覆盖场景 |
|------|------|----------|
| 1 | 直接解析 | 标准 JSON（最佳情况） |
| 2 | 截断到最后一个 `]` | 尾部多余内容 |
| 3 | 修复常见错误 | 去掉尾部逗号、给无引号 key 加引号 |
| 4 | 修复 + 截断 | 组合策略（修复后仍有尾部内容） |
| 5 | 逐个对象提取 | 最恶劣情况：只能提取碎片 |

**为什么不用更简单的方案？**
- 方案 A：只做 Zod 校验，失败就重试 → 重试成本高（每次 500ms+），且 LLM 可能重复犯同样的错误
- 方案 B：用正则提取 JSON → 无法处理嵌套结构和复杂格式
- 方案 C（当前）：5 层递进修复 → 覆盖 99%+ 的 LLM 输出问题，仅在最恶劣情况才需要重试

#### 3.1.4 去重策略

```typescript
function titleSimilar(a: string, b: string): boolean {
  const la = a.toLowerCase().trim();
  const lb = b.toLowerCase().trim();
  if (la === lb) return true;           // 完全相同
  if (la.length > 5 && lb.includes(la)) return true;  // 包含关系
  if (lb.length > 5 && la.includes(lb)) return true;
  return false;
}
```

**为什么用 title 相似度而非 content 语义相似度？**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 方案 A：基于 content cosine similarity | 语义级别去重，能识别 paraphrase | 需要调用 Embedding API，成本高，延迟大 |
| 方案 B：基于 title 相似度（当前选择） | 零成本，纯字符串比较，速度极快 | 只能识别 title 相同或包含的情况 |

**选择方案 B 的理由**：
1. AI 提炼的 title 通常高度概括，语义重复的知识往往 title 也相似
2. 去重发生在入库前，后续还有人工审核环节可以兜底
3. 如果后续需要更精确去重，可以在审核页面增加语义相似度排序

### 3.2 Prompt 工程

**文件**：`src/lib/prompts/extraction.ts`

#### 3.2.1 System Prompt 设计

```
你是一个专业的知识提炼助手。你的任务是从给定的文本中提取结构化知识条目。

输出要求：
1. 只输出 JSON 数组，不要输出任何解释
2. 不要用 ```json 代码块包裹
3. title 要简洁明确，能概括核心内容
4. content 要完整准确，保留原文的关键信息
5. suggestedCategory 是建议的分类名称
6. suggestedTags 是建议的标签列表（不超过5个）
7. 如果原文没有明确的知识点，直接返回 []
```

**为什么 Prompt 要求只输出 JSON？**
- 减少 LLM 输出的不确定性，提高解析成功率
- 避免模型输出冗长的解释性文字，浪费 token
- 便于 5 层 JSON 容错策略处理

#### 3.2.2 User Prompt 上下文增强

```typescript
function renderChunkUserPrompt(chunkContent, docName, chunkIndex, totalChunks) {
  return `[文档: ${docName}] [第 ${chunkIndex + 1}/${totalChunks} 段]\n\n${chunkContent}`;
}
```

**为什么要附加上下文信息？**
- 帮助 LLM 理解当前段落在文档中的位置
- 文档名称可能包含主题信息，有助于分类建议
- 段落位置（第几段/共几段）帮助模型判断内容的完整性

### 3.3 候选知识管理

**服务层**：`src/server/services/extraction.service.ts`

#### 核心函数

| 函数 | 职责 |
|------|------|
| `extractFromDocument(id)` | 从文档逐段提炼，返回候选知识列表 |
| `extractFromText(text)` | 从纯文本直接提炼（用于手动录入） |
| `listCandidates()` | 获取所有 pending 状态的候选知识 |
| `updateCandidate(id, data)` | 编辑候选知识（标题、内容、分类、标签） |
| `rejectCandidate(id)` | 驳回候选知识（状态改为 rejected） |
| `confirmCandidates(ids, kbIds)` | 确认入库（生成 embedding + 绑定知识库） |
| `deleteCandidateHard(id)` | 硬删除候选知识 |
| `deleteCandidatesBatch(ids)` | 批量删除 |

#### 确认入库流程

```
confirmCandidates(ids, knowledgeBaseIds):
  1. 查询 pending 状态的 knowledge chunk
  2. 确保文档已绑定到目标知识库
     → 不存在则创建 KnowledgeBaseDocument 关联
  3. 调用 indexChunks() 生成向量索引
     → 失败则回滚：删除 embedding + 恢复 disabled 状态
  4. 更新状态：
     → reviewStatus: "pending" → "confirmed"
     → chunkStatus: "disabled" → "active"
     → knowledgeBaseId: 目标知识库 ID
  5. 知识进入 RAG 可检索状态
```

**为什么确认入库时才生成 embedding？**
- 减少无效计算：审核可能拒绝的知识不需要向量化
- 降低成本：Embedding API 调用有费用
- 保证一致性：只有审核通过的知识才进入检索

### 3.4 文档管理增强

**服务层**：`src/server/services/document.service.ts`

#### 编辑后重新分段

```
用户编辑文档内容
  → PATCH /api/documents/[id] { content }
  → updateDocumentContent(id, rawContent):
      1. 内容未变 → 直接返回（幂等）
      2. 内容有变 → 重新执行分段逻辑
         → 语义分段优先 → 机械分段兜底
      3. replaceTextChunksAndIndex():
         → 只删 chunkType="text" 的旧分段
         → 保留 knowledge 类型 chunk（AI 提炼结果不丢失）
         → 写入新 text chunk
```

**关键行为**：
- 每次编辑触发完整重分段，不是增量更新
- knowledge chunk 不受影响，只替换 text chunk
- 内容完全相同则跳过（幂等）

**为什么编辑后要重分段而非增量更新？**
- 语义分段依赖全局上下文，局部修改可能影响其他段落的边界
- 重分段保证分段质量，避免增量更新导致的段落碎片化
- 幂等性保证相同内容不会产生重复操作

## 四、数据模型重构

### 4.1 重构背景

原有架构存在 **"三条平行管线"** 的数据孤岛问题：

```
变更前 (6 模型)                          变更后 (3 模型)
═══════════════                          ═══════════════
DocumentSource ──→ DocumentChunk         DocumentSource ──→ DocumentChunk
                                       (统一文档模型)     (统一分块模型)
KnowledgeDocument ──→ KnowledgeChunk          │              ├─ type="text"
                                              │              └─ type="knowledge"
CandidateKnowledge (孤岛)
```

**为什么会有三条平行管线？**
- 项目初期，文档导入、知识管理、AI 提炼分别由不同成员独立开发
- 每个模块设计了自己的数据模型，没有统一规划
- 随着功能迭代，三个模型的字段高度重叠，但无法互相访问

### 4.2 删除的模型

| 模型 | 原因 |
|------|------|
| `KnowledgeDocument` | 字段与 `DocumentSource` 高度重叠，合并到 `DocumentSource` |
| `KnowledgeChunk` | 字段是 `DocumentChunk` 的子集，合并到 `DocumentChunk` |
| `CandidateKnowledge` | AI 提炼结果死端，合并到 `DocumentChunk`（`chunkType="knowledge"`） |

### 4.3 DocumentSource 扩展字段

```diff
+ title              String         // ← KnowledgeDocument.title
+ sourceType         String         // ← KnowledgeDocument.sourceType
+ fileName           String?        // ← KnowledgeDocument.fileName
+ fileUrl            String?        // ← KnowledgeDocument.fileUrl
+ mimeType           String?        // ← KnowledgeDocument.mimeType
+ rawContent         String?        // 重命名自 content
+ chunkSize          Int            // ← KnowledgeDocument.chunkSize
+ chunkOverlap       Int            // ← KnowledgeDocument.chunkOverlap
+ activeStatus       String         // ← KnowledgeDocument.status
+ error              String?        // 统一自 errorMessage
+ knowledgeBaseId    String?        // ← KnowledgeDocument.knowledgeBaseId
```

### 4.4 DocumentChunk 扩展字段

```diff
+ documentSourceId   String?        // 改为可选（支持无来源文档的文本提炼）
+ chunkType          String         // "text" | "knowledge"
+ title              String?        // ← CandidateKnowledge.title
+ suggestedCategory  String?        // ← CandidateKnowledge.suggestedCategory
+ suggestedTags      String?        // ← CandidateKnowledge.suggestedTags (JSON)
+ reviewStatus       String?        // ← CandidateKnowledge.status
+ knowledgeType      String?        // ← CandidateKnowledge.type
+ knowledgeBaseId    String?        // ← KnowledgeChunk.knowledgeBaseId
```

### 4.5 重构收益

| 维度 | 收益 |
|------|------|
| **架构** | 消除数据孤岛，AI 提炼知识确认后自动进入 RAG 检索 |
| **开发** | 统一 CRUD，一套 API/服务层操作所有分块类型 |
| **检索** | RAG 检索一次查询覆盖原始分块 + AI 提炼知识 |
| **维护** | 6 表 → 3 表，关系复杂度大幅降低 |

**为什么用单表多态而非多表？**
- 单表避免 JOIN 查询，性能更好
- 统一的 CRUD 接口减少代码重复
- chunkType 字段可以清晰区分两种分块
- 缺点是 NULL 字段较多，但可接受

### 4.6 状态字段双轨设计

| 字段 | 语义 | 取值 |
|------|------|------|
| `status` | 处理流水线状态 | uploading → uploaded → parsing → parsed / failed |
| `activeStatus` | 业务启用状态 | active / disabled |
| `chunkStatus` | 分块启用状态 | active / disabled |
| `reviewStatus` | 审核状态 | pending → confirmed / rejected |

**为什么需要双轨设计？**
- `status` 表示文档在处理流水线中的位置（技术状态）
- `activeStatus` 表示文档是否对用户可见（业务状态）
- 两个维度独立变化：一个文档可能已解析（status=parsed）但被禁用（activeStatus=disabled）

## 七、协作接口

### 7.1 上下游关系

```
成员C(文档导入管线)
    → GET /api/documents/:id/chunks (分段数据)
    → 成员D(AI知识提炼) ← 本模块
        → 提炼后的知识 → 成员A(知识生命周期管理)

## 八、技术选型

### 8.1 LLM 调用

| 维度 | 选型 | 说明 |
|------|------|------|
| API 协议 | OpenAI Chat Completions | 兼容多种 LLM 服务 |
| 模型 | 可配置（默认 gpt-4o-mini） | 通过环境变量 `OPENAI_MODEL` |
| 温度 | 0.3 | 低温度保证输出稳定性 |
| 最大 token | 4096 | 足够输出多个知识条目 |

**为什么 temperature=0.3？**
- 知识提炼需要准确性，不需要创造性
- 0.3 保证输出稳定，减少随机性
- 0 过于死板，可能导致重复输出

### 8.2 JSON 校验

| 维度 | 选型 | 说明 |
|------|------|------|
| Schema 校验 | Zod | `CandidateKnowledgeSchema` 严格校验 |
| 容错解析 | 自研 5 层策略 | 处理 LLM 输出的各种格式问题 |

### 8.3 向量索引

| 维度 | 选型 | 说明 |
|------|------|------|
| 向量化 | DashScope text-embedding-v4 | 1024 维中文优化 |
| 存储 | ChunkEmbedding 表 | 与 text chunk 共用基础设施 |
| 索引 | indexChunks() | 统一的向量索引服务 |

## 九、关键常量说明

| 常量 | 值 | 作用 |
|------|------|------|
| `MAX_RETRIES` | 2 | LLM 调用失败重试次数 |
| `RETRY_DELAY_MS` | 500 | 重试间隔（毫秒） |
| `LLM_TEMPERATURE` | 0.3 | 生成温度（低=稳定，高=创意） |
| `LLM_MAX_TOKENS` | 4096 | 最大输出 token |
| `CANDIDATE_TITLE_MIN_LENGTH` | 5 | 去重时 title 最小长度 |

**为什么 MAX_RETRIES = 2？**
- LLM 偶尔会返回格式错误的 JSON（约 5-10% 概率）
- 重试 2 次可将成功率提升到 99%+
- 超过 2 次重试会显著增加延迟，用户体验下降

**为什么 LLM_TEMPERATURE = 0.3？**
- 知识提炼需要准确性，不需要创造性
- 0.3 保证输出稳定，减少随机性
- 0 过于死板，可能导致重复输出

## 十、风险与边界

| # | 风险 | 说明 | 应对 |
|---|------|------|------|
| 1 | LLM 输出不稳定性 | 即使 temperature=0.3，仍有 5-10% 概率返回格式错误的 JSON | 5 层 JSON 容错 + 2 次重试 |
| 2 | 去重策略的局限 | 仅基于 title 相似度，无法识别语义重复但 title 不同的知识 | 依赖人工审核环节兜底 |
| 3 | knowledge 块 documentSourceId 为空 | 从文本直接提炼的 knowledge 块没有来源文档 | 确认入库时强制绑定知识库 |
| 4 | LLM 提炼质量依赖模型能力 | 不同模型（gpt-4o-mini vs gpt-4o）效果差异大 | 通过环境变量 OPENAI_MODEL 可配置 |
| 5 | 批量提炼无进度反馈 | 长文档（100+ 分段）提炼时间长，前端无进度展示 | 后续可增加 SSE 进度推送 |
| 6 | RAG 检索过滤 knowledge 块 | documentSourceId 为空的 knowledge 块可能被过滤 | 确认入库时写入 knowledgeBaseId |

## 十一、设计决策小结

| 设计点 | 决策 | 理由 |
|--------|------|------|
| 为什么用 chunkType 区分两种块 | 单表多态 | 避免多表 JOIN，统一 CRUD |
| 为什么 knowledge 块初始 disabled | 审核后才激活 | 防止低质量知识进入检索 |
| 为什么确认入库时才生成 embedding | 延迟向量化 | 减少无效计算，审核拒绝的不生成 |
| 为什么去重用 title 而非 content | 零成本快速去重 | title 已足够区分，后续有人工审核 |
| 为什么 JSON 容错用 5 层而非 1 层 | 递进式修复 | LLM 输出问题多样，单层无法覆盖 |
| 为什么编辑后要重分段 | 保证分段质量 | 语义分段依赖全局上下文，增量更新可能导致碎片化 |
