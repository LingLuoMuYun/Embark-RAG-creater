## 写入接口（B、F 必接）

### POST /api/usage/logs

**用途：** 检索或问答完成后打一条日志。有引用 → 计热门知识；无引用 → 计知识缺口。

**谁调用：** 成员 B、成员 F

**何时调用：** `POST /api/knowledge/retrieve` 返回后（未命中也要调）

**请求体：**

```json
{
  "query": "用户问题",
  "scope": { "knowledgeBaseIds": ["kb_001"] },
  "mode": "balanced",
  "contexts": [],
  "references": []
}
```

`contexts`、`references` 直接传 B 检索返回的顶层字段，**不是** `.data.contexts`。

**示例：**

```ts
await fetch("/api/usage/logs", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    query,
    scope,
    mode: "balanced",
    contexts: retrieveResult.contexts,
    references: retrieveResult.references,
  }),
});
```

失败时 `catch` 即可，不要阻断检索/问答。

## 查询接口（只读，按需调用）

G 接口统一返回 `{ success, data, message }`。

| 接口 | 用途 | 谁用 | 前提 |
| --- | --- | --- | --- |
| `GET /api/analytics/overview` | 一次拿全量统计 | 全员 | — |
| `GET /api/analytics/hot-knowledge?limit=10` | 被引用最多的知识 | A | B/F 已打 log 且有命中 |
| `GET /api/analytics/knowledge-gaps?limit=10` | 反复未命中的问题 | A | B/F 已打 log 且有未命中 |
| `GET /api/analytics/recent-knowledge?limit=10` | 最新入库的知识 | 全员 | A 已写入知识 |
| `GET /api/analytics/category-distribution` | 各分类知识数量 | 全员 | A 给知识绑了 categoryId（当前多为「未分类」） |

## 各成员要做什么

| 成员 | 动作 |
| --- | --- |
| **B** | retrieve 后调 `POST /api/usage/logs` |
| **F** | 问答后调 `POST /api/usage/logs`；后续调 capture 沉淀知识 |
| **A** | 不用调 G；可读 hot-knowledge / knowledge-gaps 补知识；后续提供知识创建接口给 capture 用 |
| **D** | 当前不用；后续给 capture 提供 AI 提炼 |
| **E** | 不用调 G；F 读 `allowKnowledgeCapture` 控制沉淀开关 |
| **G** | 维护以上接口和 `/dashboard` |

---

## 联调顺序

```text
现在：B、F 接 usage/logs → Dashboard 有热门知识和缺口
然后：A 写知识 → recent-knowledge 有数据
然后：A 绑 categoryId → 分类分布准确
最后：F 对话 + G capture + D 提炼 + A 入库
```
# 当前版本说明

当前使用引用中的 `knowledgeId` 指向 `DocumentSource.id`，`chunkId` 指向 `DocumentChunk.id`。`UsageReference.type` 是新的 chunk 类型字段，`chunkType` 仅作为兼容字段短期保留并与 `type` 同值。
