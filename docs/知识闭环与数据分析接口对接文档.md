# 知识闭环与数据分析接口对接文档

维护人：**成员 G（知识闭环与数据分析）**

本文档说明统计模块提供的接口，以及各成员应在什么时机、如何调用这些接口。

---

## 1. 职责边界

成员 G 负责**统计和闭环**，不负责执行检索，也不负责生成回答。

上游模块在完成检索或问答后，把本次问题、检索范围、命中上下文和引用结果提交给统计接口。统计模块会根据这些数据生成：

- 热门知识排行
- 知识缺口列表
- Dashboard 总览数据
- 近 30 天文档上传热力图

**G 不会修改 B 的检索接口。** 日志由调用方在检索/问答完成后**额外** POST 到 `/api/usage/logs`。

---

## 2. 协作分工：谁需要配合、做什么

| 成员 | 模块 | 需要配合的事 | 是否必须 |
| --- | --- | --- | --- |
| **B** | 轻量知识检索 | 每次 `POST /api/knowledge/retrieve` 成功后，调用 `POST /api/usage/logs` | ✅ 必须 |
| **F** | Agent 问答 | 每次问答完成、拿到引用知识后，调用 `POST /api/usage/logs` | ✅ 必须 |
| **F** | Agent 问答 | 对话页提供「沉淀为知识」入口，调用 G 的 capture 接口（待实现） | 后续 |
| **A** | 知识生命周期 | 提供知识创建接口；G 沉淀的新知识写入 `status: pending` | 后续 |
| **D** | AI 知识提炼 | G 的 capture 流程复用 D 的 AI 提炼逻辑整理对话内容 | 后续 |
| **E** | 专家 Agent | 无需调 G 接口；F 读取 `allowKnowledgeCapture` 控制沉淀入口 | 间接 |
| **A / E** | 消费统计 | 通过 Dashboard 或 analytics 查询接口查看热门知识、缺口 | 可选 |

### 2.1 当前联调优先级

```text
第一阶段（现在）
  B、F → POST /api/usage/logs
  全员 → GET /api/analytics/* 或访问 /dashboard 查看统计

第二阶段（F 对话模块就绪后）
  F → 提供对话消息给 G
  G → POST /conversations/:id/capture
  D → 提供 AI 提炼能力
  A → 接收 pending 状态的新知识
```

## 3. 对接流程

```text
用户提问
  ↓
成员 B / 成员 F 调用 POST /api/knowledge/retrieve
  ↓
直接拿到 { query, contexts, references, llmContext, metrics? }
  ↓
成员 B / 成员 F 调用 POST /api/usage/logs 记录本次使用日志
  ↓
成员 G Dashboard（/dashboard）自动统计热门知识和知识缺口
```

> **注意：** 检索接口成功时**直接返回 RAG 响应体**，没有 `{ success: true, data: ... }` 包装。详见 B 的 `docs/知识组织检索联调说明.md` §5.3。

## 4. 数据结构

### 4.1 UsageLog（写入后返回）

```ts
type UsageLog = {
  id: string;
  source: "rag_retrieve";
  query: string;
  mode: "fast" | "balanced" | "detailed";
  scope: RagRetrieveRequest["scope"];
  hitCount: number;
  noHit: boolean;
  createdAt: string;
  references: UsageReference[];
};
```

### 4.2 UsageReference

字段与 B 检索返回的 `RagReference` 对齐。B 返回中的 `refId` 可以随请求传入，统计模块不存储该字段。

```ts
type UsageReference = {
  knowledgeBaseId: string;
  knowledgeId: string;
  chunkId: string;
  title: string;
  chunkType: "text" | "wiki" | "summary" | "qa";
};
```

### 4.3 RagContext（可选传入）

与 B 的 `POST /api/knowledge/retrieve` 返回中的 `contexts[]` 保持一致。如果只传 `contexts` 不传 `references`，统计模块会自动从 `contexts` 提取引用信息。

## 5. 写入接口：POST /api/usage/logs

记录一次 RAG 检索或 Agent 问答的使用日志。

### 5.1 调用时机

| 调用方 | 时机 |
| --- | --- |
| 成员 B | 管理端搜索/召回演示，`POST /api/knowledge/retrieve` 返回后 |
| 成员 F | Agent 问答流程中，检索完成并生成回答后 |

### 5.2 请求参数

```ts
type CreateUsageLogRequest = {
  query: string;
  scope: {
    knowledgeBaseIds: string[];       // 必填，至少一个
    knowledgeIds?: string[];
    categoryIds?: string[];
    tagIds?: string[];
    chunkTypes?: Array<"text" | "wiki" | "summary" | "qa">;
  };
  mode?: "fast" | "balanced" | "detailed";  // 默认 balanced
  contexts?: RagContext[];             // 默认 []
  references?: UsageReference[];       // 默认 []
};
```

### 5.3 请求示例（命中）

```json
{
  "query": "如何给成员配置管理员权限？",
  "scope": {
    "knowledgeBaseIds": ["kb_001"],
    "categoryIds": ["cat_permission"],

    "chunkTypes": ["text", "wiki"]
  },
  "mode": "balanced",
  "contexts": [
    {
      "id": "ctx_001",

      "chunkId": "chunk_wiki_001",
      "title": "账号权限配置说明",
      "content": "管理员可以在系统设置中进入权限管理页面...",
      "chunkType": "wiki",

  "references": [
```

## 6. 统计查询接口

G 的统计接口统一返回 `{ success: true, data: ... }`，失败返回 `{ success: false, error: { code, message } }`。

也可直接访问页面 **`/dashboard`** 查看可视化总览。

### 6.1 GET /api/analytics/overview

Dashboard 总览数据，包含：

- 文档 / 知识库 / Agent / 检索日志 / 知识缺口数量
- 最近新增文档、近 30 天上传热力图
- 热门知识 Top 5、知识缺口 Top 5

### 6.2 GET /api/analytics/hot-knowledge

热门知识排行，按引用次数降序。

```http
GET /api/analytics/hot-knowledge?limit=10
```

返回示例：

```json
{
  "success": true,
  "data": [
    {
      "knowledgeBaseId": "kb_001",
      "knowledgeId": "wiki_001",
      "title": "账号权限配置说明",
      "hitCount": 12
    }
  ],
  "message": "热门知识排行获取成功"
}
```

### 6.3 GET /api/analytics/knowledge-gaps

知识缺口列表，按同一问题未命中次数降序。

```http
GET /api/analytics/knowledge-gaps?limit=10
```

返回示例：

```json
{
  "success": true,
  "data": [
    {
      "query": "如何重置二级密码？",
      "count": 3
    }
  ],
  "message": "知识缺口获取成功"
}
```

## 7. 各成员对接指南

### 7.1 成员 B：检索完成后打日志

**你要做的事：** 在调用检索接口的地方，追加一次 usage log 写入。**不需要改检索接口本身。**

推荐封装：

```ts
import type { RagRetrieveRequest, RagRetrieveResponse } from "@/features/rag/types";

/** 调用 B 的业务侧检索入口，成功后自动记录 usage log。 */
export async function retrieveAndLogUsage(
  request: RagRetrieveRequest
): Promise<RagRetrieveResponse> {
  const retrieveResult: RagRetrieveResponse = await fetch(
    "/api/knowledge/retrieve",
    {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(request),
    }
  ).then((res) => {
    if (!res.ok) throw new Error("检索失败");
    return res.json();
  });

  // 日志写入失败不应阻断检索主流程，建议 catch 后 console.warn
  await fetch("/api/usage/logs", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      query: request.query,
      scope: request.scope,
      mode: request.mode ?? "balanced",
      contexts: retrieveResult.contexts,
      references: retrieveResult.references,
    }),
  }).catch((err) => {
    console.warn("[analytics] usage log 写入失败", err);
  });

  return retrieveResult;
}
```

**接入位置建议：**

- 管理端搜索框 `onSearch` 回调中
- 任何演示「知识召回」的前端页面
- 若 B 在后端封装了检索 service，也可在 service 层调用

**注意：**

- 检索成功响应是**顶层字段**，不是 `retrieveResult.data.contexts`
- `/api/rag/retrieve` 与 `/api/knowledge/retrieve` 返回格式相同，业务侧优先用后者
- **未命中也必须打日志**，否则 Dashboard 无法统计知识缺口

### 7.2 成员 F：问答完成后打日志

**你要做的事：** Agent 问答流程中，检索 + 生成回答完成后，用**同一次检索**的 `query`、`scope`、`contexts`、`references` 调用 `POST /api/usage/logs`。

```ts
// F 的问答流程伪代码
const retrieveResult = await retrieveKnowledge({ query, scope, mode });

// ... 组装 prompt、调用 LLM、展示回答 ...

await fetch("/api/usage/logs", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    query,
    scope,
    mode,
    contexts: retrieveResult.contexts,
    references: retrieveResult.references,
  }),
});
```

**后续配合（对话沉淀，待 G 实现 capture 接口）：**

1. 读取 Agent 配置中的 `allowKnowledgeCapture`（成员 E 提供）
2. 为 `true` 时在对话页展示「沉淀为知识」按钮
3. 点击后调用 `POST /conversations/:id/capture`（G 提供）
4. G 会调 D 的 AI 提炼 + A 的知识创建，新知识进入 `pending` 状态

## 8. 注意事项

| 规则 | 说明 |
| --- | --- |
| `query` 必填 | 用于知识缺口聚合 |
| `scope.knowledgeBaseIds` 必填 | 与 B 检索接口 scope 保持一致 |
| 未命中 | `contexts` 和 `references` 都传 `[]`，自动记为知识缺口 |
| 只传 contexts | 统计模块会从 contexts 提取引用，等价于传 references |
| references 非空 | 按 `knowledgeId` + `title` 汇总热门知识 |
| refId | B 返回的可传可不传，统计模块不存储 |
| 日志写入失败 | 建议 catch 后 warn，**不要阻断**检索/问答主流程 |
| 检索 vs 统计响应格式 | 检索接口无 success 包装；统计接口有 success 包装 |
