一、模块概览

1.0 业务背景

本项目面向企业中后台的 AI 知识库管理系统，覆盖采购、人事、财务、法务、审批、职场、安全、合规等企业场景。企业内部知识以 PDF、DOCX、XLSX、PPTX 等异构格式分散在文档和飞书中，无法直接用于 AI 知识提炼和检索。
本模块解决的核心问题是：将这些异构非结构化资料统一解析为纯文本 + 结构化分段，为下游 AI 知识提炼（成员 D）提供干净、完整、语义清晰的输入。这一环节是知识生产链路的第一道工序，后续所有 AI 提炼、检索、问答都依赖本模块的输出质量。

1.1 在系统中的位置

一句话职责：将外部非结构化资料（文件、飞书链接）解析为统一纯文本 + 结构化分段，为下游 AI 知识提炼提供较高质量输入。
浏览器上传/飞书链接 → 成员C(文档导入管线)
    → 文件存储 → 文本解析(12种格式 + 图片描述) → 语义分段(TextTiling 山谷检测)
    → 成员D(AI知识提炼) → 知识入库与检索

编码检测链（6 级回退）

编码检测链的 6 级回退顺序：jschardet 检测（置信度≥0.5用结果）→ UTF-8 → GBK → GB2312 → BIG5 → SHIFT_JIS → 兜底返回（乱码也接受）。每一步检查是否含乱码 U+FFFD，无乱码则解码成功。

表格智能识别（CSV）

CSV 文件的难点在于区分"表格数据"和"逗号分隔的普通文本"。判据：
- 至少 2 行、至少 2 列、列数一致
- 平均单元格长度 ≤ 40 字（表格数据短，段落文本长）
- 引号感知解析（"张三, 经理" → 1 个单元格，非 2 个）
宽表（>5 列）转为 key-value 格式保证可读性。

图片双层降级
Vision API 多模态描述（优先）
  → 5 角度结构化描述：文字提取→主体识别→布局→位置→禁止废话
  → 失败 → Tesseract OCR（chi_sim+eng 中英混合）
  → 全部失败 → "[图片无法识别]"
图片描述在 parseFileContent 内同步完成（DOCX/PDF/PPTX 均通过 Promise.all 并发），确保分段之前图片语义已就位，而非事后异步替换。

3.2 语义分段引擎

文件：`src/lib/semantic-splitter.ts`
依赖：DashScope text-embedding-v4, 1024 维中文向量

为什么需要语义分段

传统方案按固定字数（如 500 字）切分文档，会在句子中间截断，破坏语义完整性。语义分段的目标是在话题自然切换的位置切断，确保每个 chunk 内部语义凝聚。

当前流水线（7 步）

7 步流水线：
1. 步骤1: 结构预处理 — normalizeStructure：4种中文标题前插入 `\n\n`（### / 第X章 / 【...】/ 一、）
2. 步骤2: 细粒度拆句（保留偏移量） — splitSentences：`\n` 切行 + indexOf 记录位置 → 按 。！？.!? 拆句 → 合并孤立序号 → 合并图片描述 → 合并表格块
3. 步骤3: 向量化 — batchEmbedTexts：DashScope text-embedding-v4, 1024维, 批次大小10，仅用于计算相似度
4. 步骤4: 滑动窗口 — radius=2 低通滤波：左窗口 = avg(句_i-1, 句_i), 右窗口 = avg(句_i+1, 句_i+2), cos(左窗口, 右窗口) → sim_i
5. 步骤5: 四层断点判定：
- L1: sim < 0.45 → 绝对兜底断点
- L2: 局部极小值 + 谷深 ≥ max(0.7σ, 0.04) → TextTiling 语义断点
- L3: 标题正则匹配 → 结构强制断点
- L4: 最大句数 ≥ 15 句 → 强制断点
6. 步骤6: 长度保护：≥150字 or ≥3句 or 标题 → 允许切；否则拒绝继续累积；≥15句（L4: 句数强制）→ 强制切断
7. 步骤7: 后处理 — mergeSmallSegments（<100字合并到邻居）→ splitTextIntoChunks（>1000字二次机械切）

关键算法：TextTiling 山谷检测

算法原理
TextTiling 是 Hearst 于 1997 年提出的文本分割算法，核心思想是在相邻句子的语义相似度曲线上找山谷（局部极小值），谷底即话题切换点。

具体步骤：把文档拆成句子 → 每句用 Embedding 转成向量 → 计算相邻句对之间的余弦相似度 → 得到一条相似度曲线 → 在曲线上找山谷。

为什么要看相似度曲线？ 同一话题内的句子，语义相近，相似度维持在高原（0.85-0.95）。当话题切换时，相邻句子的语义差距突然变大，相似度急剧下降，在曲线上形成一个"断崖"——断崖底部就是山谷，即切分点。

为什么不用百分位阈值

百分位阈值法的逻辑是：对所有相邻句对按相似度从低到高排序，取第 N 分位以下的作为断点。例如 40% 分位意味着总是切掉相似度最低的 40% 句对。

在细粒度拆句背景下，这是结构性缺陷。我们的拆句引擎将文档拆成数百个句子级单元，其中 90% 以上的相邻句对是同段落内的连贯句子，相似度集中在 0.85-0.98。百分位法不管这些——它总是强制切出约 N% 的"断点"，导致同段落内被切出大量假断点，把连贯的段落切成碎片。

TextTiling 的判断依据不是全局排名，而是局部结构：只有当前位置真正形成了山谷（比左右邻居都低，且深度足够），才算断点。同段落内相似度是平坦高原——没有山谷——就不会出假断点。

滑动窗口消噪

直接比较相邻两个句子的相似度容易受单句噪声影响。例如"顺便说一下""请注意"等过渡句与前后都不相似，会产生假谷。

解决方案是滑动窗口低通滤波：用窗口半径 k=2 的左右窗口分别计算平均向量，再比较窗口间的余弦相似度。

位置 i 的相似度 sim_i = cos(
  avg(句_i-1, 句_i),      // 左窗口（2 句）
  avg(句_i+1, 句_i+2)     // 右窗口（2 句）
)

窗口平均相当于对曲线做低通滤波——单句波动被平滑，只有真正的语义趋势变化才会在 sim 上体现。k=2 是平衡点：太大会模糊断点位置，太小消噪不够。

山谷检测与谷深判断

有了相似度曲线后，逐个位置检查是否为山谷：

Step 1 — 局部极小值检查：位置 i 必须同时低于左邻居（sim_i-1）和右邻居（sim_i+1）。不满足 → 不是谷，跳过。

Step 2 — 谷深计算：找到山谷两侧的峰顶（PEAK_WINDOW=3 范围内取最大值），计算谷深：

谷深 = (左侧峰顶 + 右侧峰顶) / 2 - 谷底(sim_i)

Step 3 — 谷深阈值判定：谷深必须 ≥ `max(0.7 × σ, 0.04)` 才算真正的语义断点。

阈值由两部分组成：
- 0.7 × σ：自适应部分。σ 是整篇文档相似度的标准差——波动大的文档（话题多），σ 大，阈值自动升高，避免切太碎；均质文档 σ 小，阈值降低，敏感捕获微小话题切换。
- 0.04：绝对下限。即使文档极度均质（σ 趋近 0），也需要至少 0.04 的谷深来排除随机噪声。

四层断点体系的互补关系

单一检测机制无法覆盖所有场景，因此设计了四层互补体系：

第一层：绝对兜底（sim < 0.45）
话题渐变时——两个话题不是突然切换，而是逐步过渡——相似度持续缓慢下降但不形成明显山谷。绝对阈值 0.45 保证即使没有山谷，相似度降到足够低时也会强制切断。0.45 的取值依据：经测试，同段落内相邻句对极少低于此值（通常 ≥ 0.7），而跨话题句对通常低于此值。

第二层：TextTiling 山谷检测（局部极小值 + 谷深 ≥ max(0.7σ, 0.04)）
标准"断崖式下跌"话题切换场景。这是最主要的断点来源，覆盖 90% 以上的自然话题切换。对平坦高原不出假断点，对真山谷不遗漏。

第三层：标题强制（HEADER_PATTERN 正则匹配）
标题和正文在语义上可能高度相关——例如"## 3.2 语义分段引擎"紧随的正文大量使用"语义""分段""Embedding"等词——相似度可能高达 0.8+，山谷检测会漏过。但标题是天然的结构边界，通过正则强制在标题前切断。匹配 4 种中文标题模式：# Markdown 标题、第X章/节、【...】、一、二、三... 中文数字编号。

第四层：句数强制（≥ 15 句）
纯均质文本——如一篇全是同一主题的叙述性文章——所有相邻句对相似度都在 0.9 以上，山谷不存在，绝对阈值也不触发。如果不强制切，整篇文章会变成一个巨型 chunk，完全破坏检索精度。15 句的上限确保即使所有检测都失效，也能产生合理大小的 chunk。

文档状态机

状态流转路径：

[*] → uploading(创建记录) → uploaded(文件保存到磁盘)
    → parsing(触发解析)
        → parsed(chunks+embedding写入成功)
        → failed(解析异常)
    failed → parsing(重新解析)
    parsed → parsing(重新解析-内容变更)
    uploaded/parsed/failed → [*](删除)

- 状态即真相：前端根据 status 控制按钮显隐（解析/预览/提炼/重试）
- 幂等性：相同内容重复解析直接复用已有 chunks
- 可恢复性：failed 记录 error 信息，支持重新解析

编辑后重新分段（PATCH → updateDocumentContent）

用户在预览页修改文档内容后的完整链路：

前端预览弹窗（DocumentPreview）编辑 rawContent
  → PATCH /api/documents/[id]  { content: "修改后的全文" }
  → updateDocumentContent(id, rawContent):
      1. 内容未变 → 直接返回（无操作）
      2. 内容有变 → 重新执行全部分段逻辑:
         a. 图片文件 → 单 chunk = 新 rawContent
         b. 非图片文件 → splitTextSemantic 语义分段先试
            → 成功 → 新语义分段结果
            → 失败 → splitTextIntoChunks 机械兜底
      3. replaceTextChunksAndIndex(id, newChunks, { rawContent }):
         → 只删 chunkType="text" 的旧分段
         → 保留 knowledge 类型 chunk（AI 提炼结果不丢失）
         → 写入新 text chunk（chunkIndex 重新编号，ID 全新）
         → chunkCount 可能变化（取决于内容变化幅度）
      4. 返回更新后的完整文档

关键行为：
- 每次编辑都会触发完整重分段，不是增量更新
- knowledge chunk（AI 提炼的知识条目）不受影响，只替换 text chunk
- chunkCount、chunkIndex、chunk ID 都可能变化
- 内容完全相同则跳过（幂等），不产生数据库写操作

设计决策

- 统一解析路径：飞书导入和普通文件走同一个 `parseDocument()` 入口，语义分段优先、机械兜底。rawContent 直接写入 DB（无磁盘文件），parseDocument 检测到 rawContent 已就位后跳过文件读盘步骤。

- 兜底识别：6 种模式都未匹配但域名是 `feishu.cn` 时，从 URL 末尾提取 token（20+ 位字母数字），尝试作为 docx 类型获取内容。

3.1.2 LLM 调用流程

核心提炼函数 extractKnowledge(text):

    try:

 王硕 — Knowledge Agent与数据分析看板
1. Knowledge Agent 模块
1.1 背景与问题
  Knowledge Agent 是面向文档分析任务设计的问答能力。它更适合处理文档级筛选、跨文档对比、冲突检查、证据核对和归纳分析这类问题。
  普通 RAG 链路更偏向从知识库分片中召回局部内容，然后基于这些片段生成回答。这个方式适合直接问答，但在处理文档级任务时存在局限。例如用户希望比较多份文档、检查两份材料是否冲突，或者要求系统先判断哪些文档值得阅读，这类问题不能只依赖局部 chunk 命中。
  现有知识库机制依赖人工先把文档归到某个知识库下面，但人工分类本身可能不准确，也可能遗漏本来相关的文档。如果检索范围完全受人工预设限制，系统可能会错过真正相关的资料。因此，我设计的 Knowledge Agent 面向全局已导入且可用的文档集合工作。它会先判断哪些文档和当前问题最相关，再读取目标文档生成回答；如果文档筛选失败，再回退到 RAG 检索链路。
  这个模块要解决的核心问题是：在回答之前，先让系统具备“从全局文档中判断该读哪些文档”的能力。
1.2 技术选型与方案对比
方案 B：持久化知识地图表
  一种做法是在数据库中维护一张知识地图表，把每份文档的标题、摘要、来源、关键词、所属知识库等信息提前整理好。Knowledge Agent 查询时直接读取这张表，再让模型选择目标文档。
  这个方案的优点是查询时可以直接读取结构化数据，文档选择阶段会比较清晰。但它也会带来额外维护成本：
  - 文档导入后需要同步更新知识地图。
  - 文档重新解析后需要刷新摘要。
  - 文档删除、禁用、移动知识库后需要同步状态。
  - 如果知识地图更新不及时，Agent 看到的文档信息会和真实文档状态不一致。
  对当前项目来说，知识库、文档解析和文档状态还在迭代，单独维护一张知识地图表容易增加联调成本。
方案 C：请求时临时拼接文档摘要列表
  另一种做法是不把知识地图做成数据库表，而是在每次请求进入 Knowledge Agent 时，临时从当前可用文档中拼接一份候选文档摘要列表。执行结束后，这份列表就丢弃，不做持久化。
  我最终选择这个方案。原因是它更适合当前项目阶段：
  - 每次请求都基于当前 parsed + active 的文档生成候选列表，状态更接近真实可用文档。
  - 不需要维护额外知识地图表，减少文档导入、删除、状态变化后的同步逻辑。
  - 文档筛选逻辑集中在执行器中，方便后续调整候选数量、摘要长度和筛选策略。
  - 如果文档选择失败，还可以复用已有 RAG 链路兜底，不会影响问答可用性。
  这个方案的代价是每次请求都要重新组织候选文档信息。为了控制成本，我在实现中限制了候选文档数量、候选摘要长度、选中文档数量和最终读取正文长度。
1.3 具体实现
1.3.1 route.ts：聊天入口与模式分发
  聊天入口位于 src/app/api/chat/route.ts。这个接口负责处理聊天请求的公共流程，包括请求体校验、会话创建或复用、最近消息准备、长期记忆摘要准备，以及 SSE 输出器初始化。
  当请求中的 chatMode === "knowledge-agent" 时，路由会把执行逻辑分发给 Knowledge Agent：
  runKnowledgeAgentExecutor({
    userMessage,
    attachmentIds,
    llmInterface,
    recentMessages,
    memorySummary,
    signal,
    emit,
  })
  这样做的好处是聊天入口仍然保持统一，不需要为 Knowledge Agent 单独做一套聊天接口。普通问答、RAG 问答和 Knowledge Agent 可以共用会话、消息持久化、SSE 输出等基础能力。
1.3.2 executor.ts：核心执行器
  Knowledge Agent 的核心实现位于 src/server/services/knowledge-agent/executor.ts。执行器承担了主要业务逻辑，包括：
  1. 收集候选文档。
  2. 临时构建知识地图。
  3. 调用模型选择目标文档。
  4. 读取目标文档正文。
  5. 基于文档生成回答。
  6. 在失败时回退到 RAG。
  7. 生成引用和已使用文档信息。
  执行器输入类型：
  type RunKnowledgeAgentInput = {
    userMessage: string;
    attachmentIds?: string[];
    llmInterface?: LlmInterfaceKey;
    recentMessages: LlmMessage[];
    memorySummary: string | null;
    signal?: AbortSignal;
    emit: ChatStreamEmitter;
  };
  执行器输出类型：
  type RunKnowledgeAgentResult = {
    answer: string;
    knowledgeFiles: ChatKnowledgeFile[];
    citations: ChatCitation[];
  };
1.3.3 临时知识地图构建
  这里的“知识地图”不是数据库表，而是一次请求中临时拼接出来的候选文档摘要列表。执行器会先从全局文档集合中查询当前可用文档，筛选条件是：
  - DocumentSource.status = "parsed"
  - DocumentSource.activeStatus = "active"
  查询字段包括：
  - 文档 ID
  - 标题
  - 原始文件名
  - 文件类型
  - chunk 数量
  - rawContent
  - 前 3 个可读 chunk
  - 关联知识库 ID 和名称
  然后执行器会把每个文档整理成 CandidateDocument：
  type CandidateDocument = {
    id: string;
    title: string;
    originalName: string;
    fileType: string;
    chunkCount: number;
    snippet: string;
    knowledgeBaseIds: string[];
    knowledgeBaseNames: string[];
  };
  其中 snippet 来自 rawContent 或前几个 chunk 的截断内容。它的作用不是直接回答问题，而是帮助模型判断这份文档是否值得进一步读取。
  当前实现中使用了几个常量控制上下文规模：
  const CANDIDATE_DOCUMENT_LIMIT = 30;
  const DOCUMENT_SNIPPET_LIMIT = 800;
  const SELECTED_DOCUMENT_LIMIT = 3;
  const DOCUMENT_CONTENT_LIMIT = 12_000;
  const MIN_DOCUMENT_CONTENT_LENGTH = 120;
  这些限制分别控制候选文档数量、候选摘要长度、最终选中文档数量、文档正文长度和文档可用性下限。这样可以避免一次请求把过多文档内容塞进模型上下文。
  这份临时知识地图只存在于当前请求生命周期内。请求结束后不会保存，也不会写入数据库。
1.3.4 prompts.ts：文档选择与回答生成
  Prompt 定义位于 src/server/services/knowledge-agent/prompts.ts。当前主要拆成两个 Prompt：
  - buildKnowledgeAgentSelectorPrompt
  - buildKnowledgeAgentAnswerPrompt
  buildKnowledgeAgentSelectorPrompt 用于文档选择阶段。它会把用户问题和候选文档摘要列表交给模型，让模型返回最多 3 个目标文档 ID。
  模型需要返回严格 JSON：
  {
    "documentIds": ["id1", "id2"],
    "reason": "short explanation"
  }
  返回结果会通过 Zod 校验：
  const selectedDocumentsSchema = z.object({
    documentIds: z.array(z.string().trim().min(1)).max(3),
    reason: z.string().trim().min(1).optional(),
  });
  buildKnowledgeAgentAnswerPrompt 用于回答生成阶段。它会接收用户问题、已读取的目标文档内容和附件上下文，再让模型基于这些内容生成最终回答。
  把 Prompt 拆成“选择文档”和“生成回答”两段，可以让职责更清楚。模型第一步只判断该读哪些文档，第二步才基于正文回答问题。
1.3.5 文档读取与引用生成
  当模型返回合法的 documentIds 后，执行器会进入文档读取阶段。读取条件包括：
  - 文档 ID 必须在模型返回结果中。
  - 文档仍然满足 parsed + active。
  读取后的文档会整理成 LoadedDocument：
  type LoadedDocument = {
    id: string;
    title: string;
    chunkCount: number;
    content: string;
    supportingChunks: Array<{
      id: string;
      content: string;
      chunkType: ChatCitation["chunkType"];
    }>;
    knowledgeBaseId: string;
  };
  回答完成后，执行器会生成两类结构化结果。
  第一类是 citations，用于展示回答引用了哪些片段：
  type ChatCitation = {
    refId: string;
    chunkId: string;
    knowledgeId: string;
    documentId: string;
    knowledgeBaseId: string;
    title: string;
    content: string;
    chunkType: KnowledgeChunkType;
    score: number;
  };
  第二类是 knowledge-files，用于展示本次回答使用了哪些文档：
  type ChatKnowledgeFile = {
    id: string;
    title: string;
    chunkCount: number;
  };
  这两个结果会通过 SSE 返回前端，并在消息持久化时保存。这样用户不仅能看到回答，还能知道回答基于哪些文档和片段。
1.3.6 Fallback 设计：回退到 RAG
  Knowledge Agent 并不强制所有请求都必须完成文档选择。当前实现中保留了明确的兜底逻辑 runFallbackKnowledgeAgent。
  触发 fallback 的情况包括：
  - 没有候选文档。
  - 模型没有返回合法 JSON。
  - 模型返回的 documentIds 为空。
  - 模型返回的 documentIds 不在候选文档集合内。
  - 选中文档加载后没有足够内容。
  兜底时，系统会从候选文档关联的知识库中提取 knowledgeBaseIds，然后调用现有 RAG 检索器：
  retrieveRagContexts({
    query: userMessage,
    scope: { knowledgeBaseIds },
    mode: "balanced",
  })
  这个设计的重点是：Knowledge Agent 不是替代 RAG，而是在 RAG 前面增加文档级判断能力。文档选择成功时，它走文档阅读链路；文档选择失败时，它复用已有 RAG 链路保证回答可用。
1.3.7 SSE 与状态返回
  Knowledge Agent 执行期间会通过 SSE 向前端发送状态和追踪事件。
  关键状态包括：
  - organizing
  - reading-documents
  - generating
  - failed
  关键事件包括：
  - trace
  - status
  - token
  - citations
  - knowledge-files
  - rag-summary
  前端可以根据这些事件展示当前阶段，例如正在组织文档、正在读取文档、正在生成回答，以及最终引用了哪些文档。
1.4 风险与边界
  Knowledge Agent 更适合文档级分析任务，但它不是全知检索器。当前实现有几个边界：
  1. 候选文档数量有限。当前候选文档数量有上限，如果全局文档规模继续变大，后续需要更强的候选召回策略。
  2. 文档摘要质量会影响筛选结果。如果标题、原始内容或前几个 chunk 不能代表全文，模型可能选错文档。
  3. 文档读取有长度限制。超长文档会被截断，可能遗漏后半部分信息。
  4. 文档筛选依赖模型返回合法 JSON，所以必须保留 Zod 校验和 fallback。
  5. 当前方案不维护持久化知识地图，减少了同步成本，但每次请求都需要重新组织候选文档。

2.1 背景与问题
  数据分析看板与仪表盘模块用于后台管理场景，主要解决系统运行状态不可见的问题。
  在原有后台中，页面更像功能入口页。管理员可以进入文档管理、知识管理、Agent 对话等功能，但很难直接判断系统当前是否健康运转。例如：
  - 最近导入了多少文档。
  - 哪些知识最常被命中。
  - 哪些问题反复没有命中。
  - 当前是否存在解析失败文档。
  - 是否有待审核知识积压。
  - 哪些 Agent 最近更活跃。
  另外，检索和问答结果原本没有稳定沉淀为分析数据。检索模块知道用户问了什么，知识模块知道库里有什么，但缺少统一日志把“问题、命中、引用、未命中”记录下来。没有这些数据，后台就无法展示热门知识、知识缺口和使用趋势。
  因此，我设计了数据采集、统计聚合和页面展示三部分能力，把检索与问答行为沉淀成后台可以观察的数据。
2.2 技术选型与方案对比
方案 B：业务模块内部分散统计
  一种做法是让各业务模块自己维护统计数据。例如检索模块统计问题次数，知识模块统计知识引用次数，Agent 模块统计对话活跃度。
  这个方案看起来比较直接，但问题是统计口径会分散：
  - 每个模块都要维护自己的统计逻辑。
  - 热门知识、知识缺口、趋势数据难以统一。
  - 页面需要同时理解多个模块的数据结构。
  - 后续调整统计口径时，需要改多个模块。
  这个方案不适合当前后台首页这种总览型页面。
方案 C：统一日志记录 + 服务层聚合
  我最终选择统一日志记录和服务层聚合方案。检索或问答完成后，调用方通过 POST /api/usage/logs 写入使用日志。后台页面不直接从各业务模块拼统计，而是通过 analytics.service.ts 统一聚合。
  这个方案的优点是：
  - 数据采集和页面展示解耦。
  - 命中和未命中可以使用同一套日志模型。
  - 热门知识、知识缺口和使用趋势可以基于统一口径计算。
  - dashboard 页面只负责展示，不承担复杂统计逻辑。
  - 后续其他模块只要接入日志接口，就可以进入分析体系。
2.3 具体实现
2.3.1 数据模型：UsageLog
  UsageLog 用于记录一次检索或问答行为。
  关键字段包括：
  - id
  - source：日志来源，当前默认值为 rag_retrieve
  - query：用户查询内容
  - mode：检索模式
  - scope：检索范围
  - hitCount：命中数量
  - noHit：是否完全未命中
  - createdAt
  - updatedAt
  主要索引包括：
  - @@index([source])
  - @@index([noHit])
  - @@index([query])
  - @@index([createdAt])
  其中 noHit 支撑知识缺口统计，query 支撑高频问题聚合，createdAt 支撑使用趋势统计。
2.3.2 数据模型：UsageReference
  UsageReference 用于记录一次使用日志中命中的知识引用。
  关键字段包括：
  - id
  - usageLogId
  - knowledgeBaseId
  - knowledgeId
  - chunkId
  - title
  - chunkType
  - createdAt
  - updatedAt
  它通过 usageLogId 关联 UsageLog。删除 UsageLog 时，对应引用会级联删除。
  如果只记录“用户问了什么”，后台只能看到问题数量，看不到哪些知识真正被使用。增加 UsageReference 后，热门知识排行才有稳定的数据来源。
2.3.3 POST /api/usage/logs：日志写入接口
  文件位置：src/app/api/usage/logs/route.ts
  这个接口用于写入一次检索或问答行为日志。
  核心入参示例：
  {
    "query": "用户问题",
    "scope": {
      "knowledgeBaseIds": ["kb_001"]
    },
    "mode": "balanced",
    "contexts": [],
    "references": []
  }
  接口使用 usageLogCreateSchema 校验入参：
  - query 必填。
  - scope 必填。
  - mode 只允许 fast、balanced、detailed。
  - contexts 和 references 都支持数组输入。
  如果调用方没有传 references，服务层会尝试从 contexts 推导引用。如果 contexts 和 references 都为空，系统会把本次查询记为 noHit = true。
2.3.4 analytics.service.ts：统计聚合服务
  统计聚合逻辑集中在 src/server/services/analytics.service.ts。
  这个服务会从多个数据源聚合首页需要的信息，包括：
  - DocumentSource
  - DocumentChunk
  - KnowledgeBase
  - ExpertAgent
  - AgentConversation
  - UsageLog
  - UsageReference
  它负责拼装 overview 数据，例如总数统计、待处理事项、来源分布、使用趋势、热门知识、知识缺口、最近文档、最近知识和最近 Agent。
  页面不直接写复杂统计逻辑，而是消费服务层已经整理好的结果。这样后续调整页面布局时，不需要改动统计查询。
2.3.5 GET /api/analytics/overview：首页总览接口
  文件位置：src/app/api/analytics/overview/route.ts
  这个接口返回仪表盘首页所需的聚合数据。
  返回结构包括：
  - totals
  - pendingWorkload
  - statusBreakdown
  - recentDocuments
  - documentActivity
  - usageTrend
  - insights
  - hotKnowledge
  - knowledgeGaps
  - recentKnowledge
  - sourceDistribution
  - recentAgents
  这是 dashboard 页面最主要的数据接口。页面只请求一次 overview，就能得到大部分展示数据。
2.3.6 热门知识与知识缺口接口
  热门知识接口位于 src/app/api/analytics/hot-knowledge/route.ts，用于返回热门知识排行。它主要基于 UsageReference 聚合，按引用次数排序。
  知识缺口接口位于 src/app/api/analytics/knowledge-gaps/route.ts，用于返回高频未命中问题。它主要基于 UsageLog 中 noHit = true 的记录聚合。
  这两个接口可以单独使用，也可以作为 overview 数据的一部分进入首页展示。
2.3.7 /dashboard 页面与组件拆分
  /dashboard 页面位于 src/app/dashboard/page.tsx，是分析看板主页面。
  页面组织方式包括：
  1. 顶部问候区：根据时间输出问候语。
  2. 智能洞察区：展示当前最值得关注的信息。
  3. 知识生产活动区：展示最近导入素材和最近生成知识。
  4. Agent 区：展示最近更新或活跃的 Agent。
  5. 趋势区：展示近 7 天提问次数和新增知识变化。
  6. 来源与积压区：展示来源分布和待处理事项。
  主要组件包括：
  - insight-card.tsx
  - knowledge-production-activity.tsx
  - my-agents.tsx
  - usage-trend-chart.tsx
  - pending-workload-card.tsx
  - source-distribution-pie.tsx
  组件拆分后，页面结构更清晰。统计逻辑留在服务层，组件只负责展示。
2.3.8 智能洞察规则
  智能洞察是 dashboard 中最接近“结论层”的部分。它不直接展示原始统计值，而是把多个统计结果压缩成首页最值得先看的提示。
  当前实现位于 analytics.service.ts 的 buildDashboardInsights 方法，展示组件位于 src/features/analytics/components/insight-card.tsx。
  洞察生成依赖以下聚合数据：
  - totalDocuments
  - documentChunks
  - agents
  - failedDocuments
  - knowledgeGaps
  - pendingKnowledge
  - topAgent
  - hotKnowledge
  - usageLogs
  每条洞察包含：
  - id
  - title
  - description
  - tone
  - variant
  第一条洞察回答“当前最该先处理什么”，优先级为：
  1. failedDocuments > 0：优先提示解析失败文档。
  2. knowledgeGaps.length > 0：提示高频未命中问题。
  3. pendingKnowledge > 0：提示待审核知识。
  4. 否则提示当前没有明显阻塞项。
  第二条洞察回答“最近最值得关注什么”，优先级为：
  1. topAgent
  2. hotKnowledge
  3. usageLogs
  4. no-usage-highlight
  这样 dashboard 不是先堆图表，而是先给管理员一个结论，再用下面的图表和列表展开。
2.4 风险与边界
  数据分析看板有几个边界：
  1. 日志接入不完整会导致统计失真。如果检索或问答链路没有调用 POST /api/usage/logs，热门知识、知识缺口和使用趋势就不完整。
  2. noHit = true 只能说明当前检索结果没有命中，不能绝对说明知识库中没有相关知识。
  3. overview 是首页总览接口，不会暴露所有数据库字段，也不能替代完整管理页。
  4. 当前统计口径依赖现有 Prisma schema。如果底层文档、知识或引用模型变化，统计服务需要同步调整。
  5. 智能洞察是摘要层，不是完整告警系统。没有显示在洞察卡片中的内容，不代表系统一定没有问题。
2.5 验证方式
日志写入验证
  1. 构造一个带 references 的请求。
  2. 调用 POST /api/usage/logs。
  3. 检查返回的 hitCount、noHit 和 references。
  4. 检查数据库中是否新增 UsageLog 和对应的 UsageReference。
热门知识验证
  1. 连续写入多条包含相同 knowledgeId 的使用日志。
  2. 调用 GET /api/analytics/hot-knowledge。
  3. 确认返回列表按命中次数倒序排列。
知识缺口验证
  1. 连续写入多条 noHit = true 的问题日志。
  2. 调用 GET /api/analytics/knowledge-gaps。
  3. 确认相同问题会被合并计数。
overview 聚合验证
  1. 确认数据库中存在文档、知识、Agent、对话和日志数据。
  2. 调用 GET /api/analytics/overview。
  3. 检查 totals、pendingWorkload、statusBreakdown、usageTrend、insights、sourceDistribution 等字段是否返回。
页面展示验证
  1. 启动项目。
  2. 打开 /dashboard。
  3. 检查智能洞察、趋势图、热门知识、知识缺口、来源分布和待处理卡片是否能正常展示。
