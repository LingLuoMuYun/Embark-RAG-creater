# 聊天功能技术文档

## 1. 功能定位

聊天功能是本项目知识消费侧的统一对话入口，负责把用户问题、附件内容、历史会话、知识库检索结果、专家 Agent 配置和 Skill 能力组织起来，最终通过流式回答返回给前端。

整体目标不是单纯调用大模型，而是形成一条完整的知识消费链路：

```text
用户提问
-> 选择聊天模式
-> 创建或复用会话
-> 读取历史上下文
-> 处理附件
-> 按模式组织上下文
-> 调用模型 / RAG / Knowledge Agent / Skill Agent
-> SSE 流式返回
-> 前端打字机展示
-> 保存消息与引用
-> 按条件沉淀候选知识
```

页面入口：

```text
/agents/chat
```

## 3. 会话与页面状态

会话列表由 Zustand 管理：

```text
src/store/slices/chat-conversations-slice.ts
```

全局 store 只保存会话列表，不保存当前聊天运行态。

当前页面本地维护：

- 当前会话 ID
- 当前消息列表
- 输入框内容
- 当前聊天模式
- 当前 Agent ID
- 附件上传状态
- 是否正在生成
- 错误提示
- Skill 草稿发布状态

当前会话由 URL 参数驱动：

```text
/agents/chat?conversationId=xxx
```

打开历史会话时的流程：

```text
读取 URL conversationId
-> 从全局会话列表中查找会话
-> GET /api/conversations/[id]/messages
-> 加载历史消息
-> 重置输入框、附件、错误状态和滚动状态
```

## 4. 附件链路

聊天支持上传图片和文件。

前端上传接口：

```http
POST /api/chat/attachments
```

前端流程：

```text
用户选择文件
-> 创建本地 uploading 占位
-> 上传到 /api/chat/attachments
-> 服务端返回附件 DTO
-> 本地附件状态更新为 ready
-> 发送消息时提交 attachmentIds
```

发送消息时，只会把 `status === "ready"` 的附件提交给 `/api/chat`。

服务端会根据附件生成两类上下文：

- 文本附件：抽取文本后加入 prompt 上下文
- 图片附件：转换为模型可识别的 image parts，并注入当前用户消息

相关服务：

```text
src/server/services/chat-attachment.service.ts
src/server/services/chat/chat-prompt-helpers.ts
```

## 5. 聊天模式

统一聊天接口：

```http
POST /api/chat
```

请求体中的 `chatMode` 决定执行链路。

| chatMode | 说明 | 是否依赖知识 |
| --- | --- | --- |
| `knowledge-agent` | 默认知识助手，由 Knowledge Agent 选择知识文件并回答 | 是 |
| `openai` | 通用模型聊天，不主动检索知识库 | 否 |
| `rag-openai` | 先检索知识库，再把检索上下文交给模型 | 是 |
| `agent` | 使用专家 Agent 配置和知识范围回答 | 可选，取决于 Agent 范围 |
| `skill-agent` | 生成、保存和发布 API Skill 草稿 | 可读取知识库列表 |

## 6. 后端入口与编排

聊天接口入口：

```text
src/app/api/chat/route.ts
```

路由层职责：

```text
解析请求体
-> Zod 校验 directChatRequestSchema
-> 创建或复用 ChatConversation
-> 准备 recentMessages 和 memorySummary
-> 创建 SSE emitter
-> 调用 chat-orchestrator
-> 捕获异常并返回 error 事件
```

聊天编排服务：

```text
src/server/services/chat/chat-orchestrator.ts
```

编排服务负责连接 `/api/chat` 和实际执行器，根据 `chatMode` 组织不同 handler。

实际执行器：

```text
src/server/services/chat/chat-executor.ts
```

执行器负责：

- 判断是否进入 Skill Agent
- 检查已发布 Skill 是否可直接命中
- 调用 Knowledge Agent
- 调用 direct / RAG / Agent prompt 准备逻辑
- 调用模型流式生成
- 输出 SSE 事件
- 保存本轮用户消息和 assistant 消息
- 触发对话知识沉淀

## 7. 不同模式的执行链路

### 7.1 knowledge-agent

`knowledge-agent` 是默认知识助手模式。

执行链路：

```text
/api/chat
-> chat-orchestrator
-> chat-executor
-> runKnowledgeAgentExecutor
-> Knowledge Agent 选择并读取知识文件
-> 生成回答
-> 返回 knowledge-files / token / done
-> 保存消息
```

核心文件：

```text
src/server/services/knowledge-agent/executor.ts
```

该模式会返回 `knowledge-files`，前端可展示本轮读取过的知识文件。

### 7.2 openai

`openai` 是通用模型模式。

执行链路：

```text
/api/chat
-> chat-orchestrator
-> prepareDirectChat
-> buildPlainChatMessages
-> streamChatCompletion
-> SSE token 输出
-> 保存消息
```

该模式不主动检索知识库，`rag-summary` 状态为 `not-applicable`。

核心文件：

```text
src/server/services/chat/chat-direct.service.ts
```

### 7.3 rag-openai

`rag-openai` 是模型 + 知识库检索模式。

执行链路：

```text
/api/chat
-> shouldUseRagForMessage 判断是否需要检索
-> retrieveRagContexts 检索知识库
-> toCitations 转换引用
-> buildRagOnlyMessages 组装 prompt
-> streamChatCompletion
-> 返回 citations / rag-summary / token
-> 保存消息和引用
```

相关文件：

```text
src/server/services/chat/chat-direct.service.ts
src/server/services/rag/retriever.ts
src/server/services/rag/gating.ts
```

### 7.4 agent

`agent` 模式使用专家 Agent 的配置回答。

执行链路：

```text
/api/chat
-> 根据 agentId 查询 ExpertAgent
-> 解析 Agent knowledgeScope
-> 按范围 retrieveRagContexts
-> 将检索上下文追加到 Agent systemPrompt
-> streamChatCompletion
-> 返回 citations / token
-> 保存消息和引用
```

如果未选择 Agent，前端会直接提示用户先选择 Agent。

相关文件：

```text
src/features/agent/agent.validation.ts
src/server/services/chat/chat-direct.service.ts
```

### 7.5 skill-agent

`skill-agent` 用于创建 API Skill。

执行链路分两种。

生成草稿：

```text
/api/chat
-> chat-orchestrator
-> streamSkillAgentChat
-> 读取 active knowledge bases
-> buildSkillAgentMessages
-> streamChatCompletion
-> 返回 Skill 草稿文本
```

确认保存：

```text
用户回复 确认 / 保存 / confirm / save
-> findLatestSkillDraft 从最近对话中解析 <skill_draft>
-> skillCreateSchema 校验
-> ensureUniqueSkillSlug
-> createSkill
-> SSE 返回 skill-draft-saved
-> 前端展示 Skill 发布弹窗
```

核心文件：

```text
src/server/services/chat/chat-skill-agent.service.ts
src/app/agents/chat/_components/skill-publish-dialog.tsx
```

Skill 发布仍使用原有接口：

```http
POST /api/skills/[id]/publish
```

## 8. SSE 流式返回

`/api/chat` 返回 `text/event-stream`。

服务端 SSE emitter：

```text
src/server/services/chat/chat-stream.ts
```

前端 SSE 解析：

```text
src/app/agents/chat/_lib/chat-sse.ts
```

事件列表：

| 事件 | 说明 |
| --- | --- |
| `meta` | 返回当前 `conversationId` |
| `status` | 当前执行状态，例如 retrieving、organizing、generating |
| `trace` | 执行过程追踪，例如 Skill 检查、知识检索、生成 |
| `token` | assistant 回答增量文本 |
| `citations` | RAG 引用来源 |
| `rag-summary` | RAG 命中状态和引用数量 |
| `knowledge-files` | Knowledge Agent 读取过的文件 |
| `skill-draft-saved` | Skill 草稿已保存 |
| `error` | 错误信息 |
| `done` | 本轮回答完成 |

前端收到 `token` 后不会一次性塞入消息，而是进入 `use-typing-queue.ts`，形成逐字输出效果。

## 9. 消息持久化与记忆

会话和消息服务：

```text
src/server/services/chat-conversation.service.ts
```

主要职责：

- 创建空会话
- 根据消息创建或复用会话
- 保存用户消息
- 保存 assistant 消息
- 保存 citations
- 保存 knowledgeFiles
- 拉取历史消息
- 准备 recentMessages
- 维护 memorySummary

消息保存发生在 assistant 回答生成完成后。

历史上下文分两类：

- recentMessages：近期消息，直接加入本轮 prompt
- memorySummary：较早消息的压缩摘要，用于降低上下文长度

## 10. 对话知识沉淀

聊天链路支持从高价值对话中沉淀候选知识。

沉淀入口：

```text
src/server/services/conversation-capture.service.ts
```

触发位置：

```text
src/server/services/chat/chat-executor.ts
```

触发流程：

```text
assistant 回答生成完成
-> persistChatExchange 保存消息
-> 判断 chatMode 是否允许尝试沉淀
-> maybeCaptureFromConversation
-> 检查 agentId
-> 检查 Agent.allowKnowledgeCapture
-> 判断对话是否高价值
-> extractKnowledge 提炼候选
-> 候选去重
-> 写入 DocumentSource
-> 写入 DocumentChunk
-> 进入候选审核工作台
```

当前会尝试沉淀的模式：

```ts
["agent", "knowledge-agent", "rag-openai"]
```

真正沉淀成功还需要满足：

- 存在 `agentId`
- Agent 开启 `allowKnowledgeCapture`
- 内容符合高价值规则
- AI 成功提炼候选知识
- 候选未与已有知识重复

跳过原因包括：

| reason | 含义 |
| --- | --- |
| `missing_agent` | 没有 agentId |
| `capture_disabled` | Agent 未开启沉淀 |
| `low_value` | 内容价值不足 |
| `extract_empty` | 未提炼出候选 |
| `duplicate` | 与已有知识重复 |

沉淀结果不会直接进入正式知识库。

落库状态：

```text
DocumentSource.sourceType = "conversation"
DocumentSource.activeStatus = "disabled"
DocumentChunk.chunkType = "knowledge"
DocumentChunk.reviewStatus = "pending"
DocumentChunk.chunkStatus = "disabled"
```

候选审核入口：

```text
/candidates
```

审核通过后，候选知识才会进入可用知识链路。

## 15. 设计边界

聊天功能的几个重要边界：

- `/api/chat` 是统一聊天入口，不直接暴露多个模式接口。
- `page.tsx` 只做路由入口，聊天主体在 feature 组件中。
- 会话列表属于全局状态，当前聊天运行态属于页面局部状态。
- 附件内容只作为当前会话上下文，不自动进入知识库。
- RAG 引用会随消息保存，历史会话打开后仍可展示。
- 对话沉淀只生成待审核候选知识，不直接污染正式知识库。
- Skill Agent 生成的是 Skill 草稿，发布仍走 Skill 管理接口。
- 沉淀失败不影响聊天回答。
