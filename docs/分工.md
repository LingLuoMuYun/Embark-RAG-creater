# AI 知识库管理平台 7 人分工报告

## 1. 分工原则

本项目按照功能实现点进行分工，而不是简单按照前端、后端、测试进行分工。每位成员负责一个可独立说明、可独立演示的模块，同时覆盖前端页面、后端接口、数据模型和基础联调。

分工目标：

1. 每个人都有明确的功能产物。
2. 每个人都有一定技术含金量。
3. 每个模块都能对应项目要求中的知识生产、知识管理或知识消费能力。
4. 模块之间有清晰协作关系，避免重复开发或接口不一致。

项目整体闭环如下：

```text
知识生产 -> 知识管理 -> 轻量检索 -> 专家 Agent 问答 -> 新知识沉淀 -> 数据统计
```

## 3. 成员 A：知识生命周期管理

### 3.1 负责内容

成员 A 负责知识条目的完整生命周期，是整个系统的数据基础。

具体任务：

- 实现知识条目创建、编辑、删除、详情查看。
- 实现知识状态管理：可用、不可用、待确认。
- 实现知识来源字段展示：手动录入、文档导入、AI 提炼、对话沉淀。
- 实现知识更新时间、创建时间、使用次数等基础信息展示。
- 实现批量启用、批量停用能力。
- 设计并维护 `knowledge_items` 数据模型。
- 提供知识 CRUD API。
- 完成知识管理列表和知识编辑页面的基础交互。

### 3.2 前端产物

- 知识列表页基础结构。
- 知识详情页。
- 新建/编辑知识表单。
- 状态切换组件。
- 批量操作入口。

### 3.3 后端产物

- `POST /knowledge`
- `GET /knowledge`
- `GET /knowledge/:id`
- `PATCH /knowledge/:id`
- `DELETE /knowledge/:id`
- `PATCH /knowledge/:id/status`
- `PATCH /knowledge/batch/status`

### 3.4 需要协作的人

- 与成员 B 协作：知识列表需要接入分类、标签和检索条件。
- 与成员 D 协作：AI 提炼出的候选知识需要进入待确认状态。
- 与成员 F 协作：Agent 回答引用知识时，需要能读取知识详情。
- 与成员 G 协作：对话沉淀的新知识需要复用知识创建接口。

### 3.5 答辩表述

我负责知识条目的生命周期管理，包括创建、编辑、删除、状态流转和来源追踪。系统中的 Agent 只能使用可用状态的知识，AI 生成或对话沉淀的新知识会先进入待确认状态，避免错误知识直接污染知识库。

## 4. 成员 B：轻量知识检索与组织

### 4.1 负责内容

成员 B 负责知识的组织、分类、标签和轻量检索能力。该模块同时服务管理端搜索和 Agent 问答前的候选知识召回。

具体任务：

- 实现分类管理：创建、编辑、删除、列表。
- 实现标签管理：创建、绑定、移除、列表。
- 实现知识列表的关键词搜索。
- 实现分类筛选、标签筛选、状态筛选。
- 实现搜索结果高亮。
- 实现最近搜索记录。
- 实现 Agent 问答前的候选知识检索接口。
- 当候选知识较多时，接入 LLM 相关性重排。
- 保证检索只返回可用知识给 Agent。

### 4.2 前端产物

- 分类管理入口。
- 标签管理入口。
- 知识搜索框。
- 分类、标签、状态筛选器。
- 搜索结果高亮展示。

### 4.3 后端产物

- `POST /categories`
- `GET /categories`
- `PATCH /categories/:id`
- `DELETE /categories/:id`
- `POST /tags`
- `GET /tags`
- `DELETE /tags/:id`
- `GET /knowledge/search`
- `POST /knowledge/retrieve`
- `POST /knowledge/rerank`

### 4.4 需要协作的人

- 与成员 A 协作：知识条目需要绑定分类和标签。
- 与成员 D 协作：AI 提炼知识时需要自动生成建议标签和分类。
- 与成员 E 协作：Agent 配置知识范围时需要选择分类、标签或指定知识。
- 与成员 F 协作：Agent 问答依赖检索模块获取候选知识。
- 与成员 G 协作：搜索和引用日志需要进入统计模块。

### 4.5 答辩表述

我负责轻量知识检索模块。系统第一版不使用重型向量数据库，而是通过关键词、分类、标签和状态过滤先召回候选知识，再使用 LLM 对候选知识进行相关性重排，保证 Agent 能找到更相关的知识内容。

## 5. 成员 C：文档导入管线

### 5.1 负责内容

成员 C 负责文档导入与解析，将外部非结构化资料转化为后续 AI 提炼可使用的文本材料。

具体任务：

- 实现文档上传。
- 支持 TXT、Markdown、DOCX、文本型 PDF。
- 实现文件类型和大小校验。
- 保存文档来源记录。
- 实现文档解析状态：上传中、解析中、已解析、失败。
- 实现文档导入记录列表。
- 实现解析文本预览。
- 实现重新解析和删除导入记录。
- 设计并维护 `document_sources` 数据模型。

### 5.2 前端产物

- 文档上传页面。
- 上传进度和解析状态展示。
- 导入记录列表。
- 解析文本预览弹窗或详情页。
- 重新解析按钮。

### 5.3 后端产物

- `POST /documents/upload`
- `GET /documents`
- `GET /documents/:id`
- `POST /documents/:id/parse`
- `DELETE /documents/:id`

### 5.4 需要协作的人

- 与成员 D 协作：解析后的文本需要交给 AI 提炼模块生成候选知识。
- 与成员 A 协作：文档最终生成的知识条目需要写入知识表。
- 与成员 G 协作：文档生成知识后，可以统计知识来源。

### 5.5 答辩表述

我负责文档导入管线，将 TXT、Markdown、DOCX 和文本型 PDF 等非结构化文件解析为统一文本，并用文档状态机管理上传、解析、失败和重试流程，为 AI 知识提炼模块提供稳定输入。

## 6. 成员 D：AI 知识提炼与审核流

### 6.1 负责内容

成员 D 负责知识生产中的 AI 自动提炼能力，是项目的核心 AI 模块之一。

具体任务：

- 提供候选知识预览。

- 支持批量确认保存。
- 支持拒绝候选知识。
- 提炼失败时支持重试。
- 生成的知识默认进入待确认状态。

### 6.2 前端产物

- AI 提炼入口。
- 粘贴材料提炼页面。
- 候选知识预览列表。
- 候选知识编辑面板。
- 批量确认、拒绝、重试操作。

### 6.3 后端产物

- `POST /ai/extract/from-text`
- `POST /ai/extract/retry`
- `POST /knowledge/candidates/confirm`

### 6.4 需要协作的人

- 与成员 C 协作：从文档导入模块接收解析文本。
- 与成员 A 协作：确认后的候选知识写入知识条目表。
- 与成员 B 协作：AI 提炼时生成建议分类和标签。
- 与成员 G 协作：对话沉淀知识时可复用 AI 提炼能力。

### 6.5 答辩表述

我负责 AI 知识提炼模块，通过 Prompt 将文档或文本材料转换为结构化知识条目，并使用 Zod 对模型输出进行结构校验。AI 生成的知识不会直接进入可用状态，而是先进入审核流程，保证知识库质量。

## 7. 成员 E：专家 Agent 配置与生成

### 7.1 负责内容

成员 E 负责将知识库包装成可消费的专家 Agent，完成从知识管理到智能服务的连接。

具体任务：

- 设置 Agent 名称和描述。

- 设置 Agent 知识范围：全部知识、指定分类、指定标签、指定知识条目。
- 设置是否展示引用。
- 设置是否允许对话沉淀新知识。
- 生成 Agent system prompt。
- 实现 Agent 可用性检查：是否绑定了可用知识。
- 设计并维护 `expert_agents` 数据模型。

### 7.2 前端产物

- Agent 列表页。
- Agent 创建/编辑页面。
- 知识范围选择器。
- 回答风格选择器。
- Agent 预览卡片。
- 进入对话按钮。

### 7.3 后端产物

- `POST /agents`
- `GET /agents`
- `GET /agents/:id`
- `PATCH /agents/:id`
- `DELETE /agents/:id`
- `POST /agents/:id/validate`
- `POST /agents/:id/system-prompt`

### 7.4 需要协作的人

- 与成员 F 协作：对话模块需要读取 Agent 配置和 system prompt。
- 与成员 G 协作：Agent 是否允许沉淀新知识会影响闭环模块。

## 10. 模块协作关系

### 10.1 核心依赖链

```text
成员 A：知识生命周期管理
成员 B：分类标签与检索
    ↓
成员 C：文档导入管线
成员 D：AI 知识提炼
    ↓
成员 E：专家 Agent 生成
成员 F：Agent 问答消费
    ↓
成员 G：知识沉淀与统计
```

### 10.2 关键协作点

1. 成员 A 和成员 B 需要先确定知识条目、分类、标签的数据结构。
2. 成员 C 输出解析文本，成员 D 基于解析文本生成候选知识。
3. 成员 D 生成的候选知识，需要通过成员 A 的知识接口保存。
4. 成员 E 创建 Agent 时，需要调用成员 B 的分类、标签和知识范围能力。
5. 成员 F 对话问答时，需要读取成员 E 的 Agent 配置，并调用成员 B 的检索能力。
6. 成员 F 产生的对话结果，需要交给成员 G 做知识沉淀和统计。
7. 成员 G 生成的新知识，需要回到成员 A 的待确认知识流程。

## 12. 接口协作约定

为了减少联调成本，所有成员应遵守以下约定：

1. 后端接口返回统一格式。

```json
{
  "success": true,
  "data": {},
  "message": ""
}
```

2. 失败时返回统一错误结构。

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "参数错误"
  }
}
```

3. 知识状态统一使用英文枚举。

```text
available
disabled
pending
```

4. 所有 AI 生成内容默认进入 `pending` 状态，必须经过确认后才能进入 `available` 状态。

5. Agent 问答只能使用 `available` 状态的知识。

6. 需要跨模块调用的数据结构，优先由模块负责人先写 mock 数据和接口文档。

## 13. 分支协作建议

建议使用以下 Git 分支：

```text
main
dev
feature/knowledge-lifecycle
feature/knowledge-retrieval
feature/document-import
feature/ai-extraction
feature/agent-builder
feature/agent-chat
feature/analytics-loop
```

协作规则：

1. `main` 保持可演示版本。
2. `dev` 用于日常集成。
3. 每个成员在自己的 feature 分支开发。
4. 每个功能完成后提交 Pull Request 合并到 `dev`。
5. 合并前至少完成本模块自测和一次前后端联调。
6. 每周或每个阶段结束后，将稳定的 `dev` 合并到 `main`。

## 14. 每人最终交付物

每位成员最终都需要提交：

1. 对应模块的前端页面或组件。
2. 对应模块的后端 API。

## 15. 总结

该分工方案将 7 名成员分别绑定到 7 个功能实现点，每个功能都对应项目核心要求或加分项。相比按前端、后端、测试简单拆分，该方案更能保证每个人都有明确贡献，也更方便在答辩中说明个人负责内容。

整体上，成员 A 和 B 负责知识管理地基，成员 C 和 D 负责知识生产，成员 E 和 F 负责知识消费，成员 G 负责闭环和可视化加分项。七个模块共同组成一个完整的 AI 知识库管理平台。
